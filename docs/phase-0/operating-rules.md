## กติกาบันทึกหน้างาน

1. รอบส่งใช้จัดกลุ่มพนักงานและรายการขายในช่วงเวลา ไม่ใช่เส้นทางหรือคลังสต๊อก และไม่มีจำนวนรอบมาตรฐานต่อวัน. เมื่อเปิดรอบ ระบบคัดลอกร้านที่เปิดใช้งานทั้งหมดเป็น `round_stops` พร้อมรหัส ชื่อ อาคาร ชั้น/โซน และลำดับ ณ เวลานั้น เพื่อให้ประวัติรอบไม่เปลี่ยนตามข้อมูลร้านภายหลัง; พนักงานเป็นผู้เลือกร้านที่จะไปส่งเอง.
2. การส่งหนึ่งครั้งเป็น `delivery_event` หนึ่งรายการ มีน้ำแข็งได้หนึ่งหรือหลายประเภทใน `delivery_items` และจำนวนของแต่ละประเภทต้องเป็นจำนวนเต็มมากกว่า 0; ร้านเดียวบันทึกได้หลายครั้งในรอบและหลายรอบในวันเดียวกัน.
3. เมื่อบันทึกรายการส่งสำเร็จ สถานะร้านของรอบนั้นเป็น `delivered` เว้นแต่หัวหน้าหรือพนักงานเลือกสถานะปัญหาในภายหลัง.
4. สถานะ `full_bin`, `closed_shop`, `no_access`, `issue` ต้องมีหมายเหตุ. สถานะ `pending` ไม่ใช่ปัญหา แต่หมายถึงยังไม่มีการดำเนินการ.
5. การยกเลิกเปลี่ยนสถานะ `delivery_event` เป็น `cancelled` พร้อมผู้ยกเลิก เวลา และเหตุผล; รายการน้ำแข็งทุกชนิดใต้เหตุการณ์นั้นไม่ถูกนับในยอด และห้ามลบแถว.
6. ทุกการแก้ไขและยกเลิกต้องเพิ่ม `audit_logs` พร้อมค่าเดิม/ใหม่และเหตุผลเมื่อกฎบทบาทกำหนด.
7. การปิดรอบอนุญาตแม้ยังมี `pending` หรือยอดต่าง. การปิดต้องบันทึก `round_close_summaries` และ `round_close_ice_summaries` ใน transaction เดียวกับการเปลี่ยนรอบเป็น `closed` เพื่อให้ยอด ณ เวลาปิดไม่เปลี่ยนย้อนหลัง.
8. หลังปิดรอบ ห้ามแก้ `round_stops`, `round_ice_counts` หรือรายการส่งโดยตรง. การแก้ย้อนหลังต้องเป็นกระบวนการปรับปรุงที่สร้าง audit log และไม่เขียนทับ snapshot เดิม.

## กติกาสต๊อกเส้นทางประจำวัน

1. รถบรรทุกเป็นคลังกลางเคลื่อนที่ของวัน เส้นทางหลักคือ `โรงงาน → Skywalk → ตึก A → ตึก B → ตึก C`; รถเล็กอาจวนกลับไปเติม Skywalk หรือตึก A ได้หลายครั้ง.
2. ก่อนเริ่มงาน หัวหน้าต้องบันทึกรายการสั่งจากโรงงานโดยระบุชนิดและจำนวนทุกครั้งที่สั่ง; ยอดสั่งเป็นสต๊อกรถตั้งต้นตามระบบ. การสั่งเพิ่มระหว่างวันต้องเป็นรายการใหม่พร้อมเวลา ชนิด และจำนวน.
3. ทุกการจ่ายน้ำแข็งให้พนักงานหรือรถเล็กต้องมีผู้รับ จุดถือครองปลายทาง ชนิด และจำนวน. ยอดที่จ่ายเป็นยอดส่งมอบ ไม่ใช่ยอดขาย.
4. หัวหน้าบันทึกยอดที่ยกลงจากรถไปยังจุด Skywalk และตึก A ตามจำนวนที่ตรวจนับจริง ก่อนจ่ายจากจุดนั้นให้พนักงาน. ถังสำรองของแต่ละจุดเป็นจุดถือครองแยกต่างหาก.
5. ที่ตึก B ต้องแยกการโอนอย่างน้อยเป็นพนักงานตึก B แต่ละคน และรถเล็ก/พนักงานตึก C. ผู้รับทุกคนเป็นจุดถือครองแยกกันจนกว่าจะบันทึกคืนหรือโอนต่อ.
6. เมื่อพนักงานหรือรถเล็กกลับมารายงาน หัวหน้าต้องกรอกยอดเหลือจริงแยกตามชนิดและเทียบกับยอดตามระบบ จากนั้นบันทึกยอดคืนรถ รับเพิ่ม โอนเข้าถังสำรอง โอนไปจุดอื่น หรือเสียหายเป็นรายการใหม่เสมอ ห้ามแก้ทับยอดโอนเดิม.
7. การขายร้านหักจากจุดต้นทางที่ระบุในรายการขาย; การย้ายระหว่างรถ จุดปฏิบัติงาน พนักงาน รถเล็ก และถังสำรองไม่ใช่ยอดขายและไม่ลดสต๊อกรวมของวัน.
8. ปิดรอบส่งไม่ปิดสต๊อกของวัน การนับจริง เทียบส่วนต่าง และส่งของเหลือกลับโรงงานเกิดหลังจบงานทั้งวันเท่านั้น.
9. การปิดรอบต้อง snapshot ยอดสต๊อกของทุกจุด ณ เวลาปิดเพื่อใช้เป็นประวัติอ่านอย่างเดียว; snapshot นี้ไม่ปิด ledger ของวันและรายการภายหลังยังแสดงในมุมมองสต๊อกตาม `service_date`.
10. การปิดสต๊อกสิ้นวันใช้ผลตรวจล่าสุดของทุกจุดเฉพาะเมื่อไม่มีรายการสต๊อกเกิดตามหลัง; จุดที่ยังไม่ตรวจหรือผลตรวจเก่าต้องตรวจใหม่ หรือให้หัวหน้ายืนยันใช้ยอดตามระบบสำหรับวันนั้น.

## ข้อมูลบนบัตรร้าน

- รูปใช้ `shops.image_path` ซึ่งอ้าง path ถาวรในพื้นที่เก็บไฟล์ ไม่เก็บ signed URL ที่หมดอายุ.
- `payment_status` เดิมเป็นเพียงธงหน้างาน (`unknown`, `paid`, `unpaid`) และต้องเลิกใช้เป็นแหล่งจริงเมื่อระบบรับชำระเริ่มใช้งาน; สถานะชำระต้องคำนวณจากยอดเอกสารลบด้วยยอดรับชำระที่จัดสรรแล้ว.
- ร้านต้องมี `payment_term` แยกจากสถานะชำระ: `daily_collection` สำหรับเก็บเงินในรอบสุดท้าย หรือ `credit` สำหรับวางบิลและรับชำระภายหลัง.
- บัตรร้านแสดงเงื่อนไขชำระและยอดค้างที่ผู้ใช้มีสิทธิ์เห็น แต่พนักงานห้ามแก้เงื่อนไขเครดิตหรือราคาเอง.
- ยอดส่งล่าสุดคือ `delivery_event` สถานะ `active` ที่มี `recorded_at` ล่าสุดของร้าน.
- ยอดเฉลี่ยรายประเภทคำนวณจากยอด `delivery_items` สถานะใช้งานย้อนหลัง 30 `service_date`; วันที่ร้านอยู่ใน `round_stops` แต่ไม่มีรายการของประเภทนั้นนับเป็น 0.
- จำนวนรอบปกติต่อวันเก็บใน `shops.normal_rounds_per_day` และใช้เพื่อแสดงข้อมูลอ้างอิงเท่านั้น ไม่สร้างรอบอัตโนมัติ.

## Transaction ที่ต้องเป็น atomic

1. เปิดรอบ: สร้าง `delivery_rounds`, สมาชิก, `round_stops` จากทุกร้านที่เปิดใช้งาน และ `round_ice_counts` ให้สำเร็จทั้งหมดหรือย้อนกลับทั้งหมด.
2. บันทึกส่ง: ตรวจว่ารอบยังเปิด ผู้ใช้เป็นสมาชิกรอบ และ `round_stop` อยู่ในรอบ จากนั้นสร้าง `delivery_event`, `delivery_items`, เปลี่ยนสถานะร้าน และเขียน audit log ให้สำเร็จทั้งหมดหรือย้อนกลับทั้งหมด. ต้องปฏิเสธเหตุการณ์ที่ไม่มี item.
3. ปิดรอบ: อ่านยอด live, บันทึก snapshot ทั้งสองตาราง, เปลี่ยนสถานะรอบ และเขียน audit log ให้สำเร็จทั้งหมดหรือย้อนกลับทั้งหมด.
4. แก้ไข/ยกเลิก: เปลี่ยนข้อมูลและสร้าง audit log ใน transaction เดียวกันเสมอ.
5. สร้างสรุปยอดสิ้นวัน: เลือกเฉพาะรายการส่งสถานะใช้งานของร้านและวันที่เดียวกัน บันทึกราคา snapshot และผูกที่มาของทุกรายการให้สำเร็จทั้งหมดหรือย้อนกลับทั้งหมด; เรียกซ้ำต้องได้เอกสารเดิม.
6. รับชำระและออกใบเสร็จ: สร้างรายการรับเงิน จัดสรรยอดให้เอกสาร อัปเดตยอดคงค้าง ออกเลขที่ใบเสร็จ และเขียน audit log ใน transaction เดียวกัน; ห้ามถือว่าการกดพิมพ์คือการรับเงิน.

## กติกาสิ้นวัน เครดิต และเอกสาร

1. “สิ้นวัน” หมายถึงเมื่อปิดรอบส่งที่กำหนดเป็นรอบสุดท้ายของวัน หรือหัวหน้า/แอดมินยืนยันว่าการส่งของวันนั้นเสร็จแล้ว ไม่ใช่อาศัยเวลาบนนาฬิกาเพียงอย่างเดียว.
2. ร้าน `daily_collection` ถูกเพิ่มในรายการเก็บเงินรอบสุดท้ายหลังมีสรุปยอดของวัน ร้าน `credit` ไม่ถูกเพิ่มในรายการนี้ เว้นแต่มีใบวางบิลครบกำหนดและแอดมินมอบหมายให้เก็บ.
3. เอกสารก่อนรับเงินจริงเรียก “ใบสรุปยอด/ใบแจ้งยอด”; “ใบเสร็จรับเงิน” สร้างหลังบันทึกรับเงินจริงเท่านั้น. หากรับบางส่วน ใบเสร็จแสดงยอดที่รับจริงและเอกสารต้นทางยังมียอดคงค้าง.
4. ราคาในรายการส่งเป็น snapshot จากราคาพิเศษของร้านที่มีผล ณ เวลาส่ง หรือราคามาตรฐานหากไม่มีราคาพิเศษ. การเปลี่ยนราคาใหม่ห้ามแก้ยอดเดิม.
5. เอกสารที่ออกเลขแล้วห้ามลบหรือแก้ยอดโดยตรง. การแก้ต้องยกเลิกพร้อมเหตุผลและออกเอกสารทดแทน โดยเก็บความเชื่อมโยงและ audit log.
6. การพิมพ์ซ้ำใช้เลขที่เอกสารและข้อมูลเดิม พร้อมระบุว่าเป็นสำเนาได้ แต่ห้ามสร้างยอดขาย ลูกหนี้ หรือรายการรับเงินใหม่.
7. ต้องแยก `ยอดขาย`, `ยอดรับเงินจริง`, และ `ยอดคงค้าง` ออกจากกันเสมอ เพื่อให้ร้านเครดิตและการรับเงินบางส่วนตรวจสอบย้อนหลังได้.

## ออฟไลน์และซิงก์ซ้ำ

Phase 4 จะทำ PWA เต็มรูปแบบ แต่ contract นี้ต้องใช้ตั้งแต่จุดเริ่มต้นเพื่อไม่ให้ข้อมูลซ้ำในอนาคต.

1. เครื่องสร้าง UUID หนึ่งค่าเป็น `idempotency_key` ต่อการกดยืนยันหนึ่งครั้ง และเก็บไว้กับ `delivery_event` ในคิวก่อนส่ง.
2. การส่งซ้ำด้วย key เดิมต้องตอบกลับเหตุการณ์เดิมพร้อมรายการน้ำแข็งเดิม ไม่สร้าง `delivery_events` หรือ `delivery_items` ซ้ำ และตอบกลับได้ต่อเมื่อผู้ขอยังมีสิทธิ์เห็นรอบนั้น.
3. ไคลเอนต์ retry ได้เฉพาะรายการที่ยังไม่มีผลตอบรับสำเร็จ; ห้ามสร้าง key ใหม่เพียงเพราะการเชื่อมต่อขาด.
4. ข้อมูลต้นทางที่เก็บออฟไลน์ใช้เวลาจากเครื่องใน `client_recorded_at`; เซิร์ฟเวอร์บันทึกเวลารับจริงใน `recorded_at`.
5. ถ้า `round_stop` ไม่มีอยู่ รอบถูกปิด หรือผู้ใช้ไม่ใช่สมาชิกรอบก่อนซิงก์ เซิร์ฟเวอร์ต้องปฏิเสธรายการพร้อมเหตุผลที่แก้ได้ และคงรายการในคิวให้ผู้ใช้ตรวจ ไม่ทิ้งเงียบ ๆ.
6. การแก้ไขและยกเลิกต้องใช้คำขอที่มีรหัสรายการเป้าหมายและตรวจสิทธิ์บนเซิร์ฟเวอร์ทุกครั้ง; ห้ามใช้ข้อมูลบทบาทจากเครื่องเป็นหลักฐานสิทธิ์.
