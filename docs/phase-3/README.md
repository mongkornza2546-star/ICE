# Phase 3: ส่วนหัวหน้า (manager first)

สถานะ: เริ่มพัฒนา โดยพักการขยายหน้าพนักงานไว้ก่อนตามลำดับใช้งานจริง

## ทำแล้ว

- หน้าหัวหน้าเป็นหน้าเริ่มต้นของ `round_lead` และ `admin`
- ภาพรวมร้านทั้งหมด ร้านที่มีรายการ ร้านที่ยังไม่มีรายการ และร้านมีปัญหา
- ตรวจยอดแยกชนิดจาก `ยกออก + เติม − เหลือ − เสียหาย` เทียบกับรายการส่งจริง
- ปิดรอบแบบ atomic และเก็บ snapshot ร้าน/น้ำแข็งผ่าน migration `0004_manager_round_control.sql`
- ตั้งค่าข้อมูลหลักตามลำดับ `ตึก → โซนย่อย → ร้าน`
- migration `0005_building_zones.sql` ย้ายค่าโซนข้อความเดิมเป็น `building_zones`, บังคับร้านเลือก `zone_id`, และป้องกันร้านชี้ข้ามตึก
- หน้าแอดมินแยกเมนู “ตึกและโซนย่อย” ก่อนเมนู “ร้านค้า”; หน้าร้านแสดงเฉพาะโซนของตึกที่เลือก
- การบันทึกร้านใช้ RPC `save_shop` transaction เดียว; ร้านที่เปิดใช้งานจะถูกเพิ่มเข้าในรอบส่งใหม่โดยอัตโนมัติ และไม่ต้องกำหนดเส้นทางหรือลำดับร้าน
- `record_delivery` และ `close_delivery_round` lock แถวรอบเดียวกัน เพื่อไม่ให้รายการส่งพร้อมการปิดรอบหลุดจาก snapshot
- หน้าควบคุมรอบปฏิเสธ summary response เก่าหลังผู้ใช้สลับรอบ
- เพิ่ม ledger สต๊อกต่อเนื่องทั้งวันใน migration `0007_daily_mobile_stock.sql` แยกยอดตามรถบรรทุก จุดปฏิบัติงาน ทีม รถเล็ก จุดหน้ารถ และถังสำรอง
- หัวหน้าบันทึกยอดสั่งจากโรงงาน โอนระหว่างจุด ของเสีย/ละลาย และส่งคืนโรงงานผ่าน RPC `record_stock_movement` โดยรายการเดียวรองรับหลายชนิดน้ำแข็งและ idempotency key
- ยอดสต๊อกของวันรวมการเคลื่อนไหวทุก round และหักรายการขายจากจุดต้นทางที่ snapshot ไว้กับ `delivery_events` อัตโนมัติ
- หน้า “สต๊อกต่อเนื่องทั้งวัน” แสดงยอดคงเหลือทุกจุด แจ้งยอดติดลบ และประวัติรายการล่าสุดบนมือถือ
- `record_delivery` และ `record_stock_movement` ใช้ lock ตามวันที่ให้บริการชุดเดียวกัน และปฏิเสธรายการที่ทำให้จุดต้นทางมีสต๊อกไม่พอ แม้เกิดจากหลาย round พร้อมกัน
- รายการส่งที่มีอยู่ก่อน migration `0007` ไม่ถูกเดาจุดต้นทางย้อนหลัง จึงไม่สร้างยอดติดลบจากข้อมูลที่ไม่มี movement ตั้งต้น
- หลังปิดรอบยังบันทึกการโอนกลับ ของเสีย และส่งคืนโรงงานเพื่อปิดงานของวันได้; การสั่งเพิ่มจากโรงงานต้องมีอย่างน้อยหนึ่งรอบเปิดในวันนั้น

## ขอบเขตที่ยังเหลือของส่วนหัวหน้า

- หน้าจัดการเพิ่ม/แก้ไขจุดถือครองประเภททีม รถเล็ก จุดหน้ารถ และถังสำรอง (migration รองรับชนิดข้อมูลแล้ว; ตอนนี้สร้างรถหลักและจุดปฏิบัติงานของแต่ละตึกให้อัตโนมัติ)
- ตรวจนับจริงสิ้นวัน ส่งคืนโรงงาน และปิดสต๊อกทุกจุดเป็นศูนย์
- แก้ไข/ยกเลิกรายการส่งของทุกคนพร้อมเหตุผล
- ราคา เงื่อนไขชำระ สรุปยอด เก็บเงิน และพิมพ์เอกสารใน Phase 3.5

## การตรวจสอบล่าสุด

- `npm run build` ผ่าน
- `npm test` ผ่าน 11 static regression checks รวม day-wide locking ของ delivery/movement, historical cutover, final movement หลังปิดรอบ และ retry idempotency
- workspace มี frontend env แล้ว แต่ยังไม่ได้ apply migration `0007` กับ Supabase project จริงเพราะไม่มี Supabase CLI/project link
- ยังไม่ได้รัน PostgreSQL integration test ในเครื่อง เพราะ Docker daemon ไม่ทำงาน และเครื่องนี้ไม่มี Supabase CLI
