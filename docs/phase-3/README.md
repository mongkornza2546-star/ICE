# Phase 3: งานหัวหน้าและแอดมิน

สถานะ: migrations `0001`–`0008` และ frontend ผ่านการทดสอบรับงานด้วยบัญชี `admin` และ `round_lead` จริงบน Supabase เมื่อ 15 กรกฎาคม 2569; migrations `0009`–`0015` อยู่ใน workspace พร้อม regression ที่เกี่ยวข้อง และ migrations จนถึง `0015` ยังรอ apply/ทดสอบกับ Supabase project จริง

## หน้าที่เปิดใช้งาน

- `round_lead` และ `admin` เริ่มที่แดชบอร์ดงานวันนี้ ซึ่งโหลดรอบส่ง ความคืบหน้าร้าน ปัญหาที่แจ้ง ยอดสต๊อกตามจุด สถานะการตรวจนับ และสถานะปิดวันจาก Supabase พร้อมทางลัดไปงานถัดไป
- แยกงานหัวหน้าเป็นหน้าชัดเจน: สั่งน้ำแข็งจากโรงงาน, ควบคุมรอบส่ง, บันทึกส่งน้ำแข็ง, โอน/ตรวจ/ปิดสต๊อก และตั้งค่าจุดถือครองสต๊อก
- หน้าสั่งน้ำแข็งโหลดชนิดน้ำแข็งและรถบรรทุกจริง แล้วบันทึกผ่าน `record_factory_order(service_date, truck_location_id, ...)` พร้อม idempotency key และประวัติจาก stock ledger โดยไม่ต้องมีรอบส่ง ไม่มีราคา เครดิต เส้นทาง หรือประวัติจำลอง
- หน้าควบคุมรอบรองรับเปิดรอบ เลือกสมาชิก ดูสรุปรอบ แก้ไขหรือยกเลิกรายการส่งพร้อมเหตุผล และปิดรอบ
- หน้าสต๊อกรองรับโอนระหว่างจุด บันทึกของเสีย/ละลาย ส่งคืนโรงงาน ตรวจนับยอดจริง และปิดสต๊อกสิ้นวัน โดยแยกออกจากหน้าตั้งค่าจุดถือครอง
- หน้าแอดมินยังจัดการตึก/โซนย่อยและร้านได้ และมีหน้าข้อมูลอ้างอิงสำหรับแก้โปรไฟล์ บทบาท และสถานะของผู้ใช้ที่มีบัญชีอยู่แล้ว รวมถึงเพิ่ม/แก้/พักใช้ชนิดน้ำแข็ง หน้านี้ไม่สร้าง Supabase Auth user หรือรหัสผ่านใหม่
- `round_lead` ใช้งานส่วนปฏิบัติการได้ ส่วนการจัดการตึก ร้าน ผู้ใช้ และชนิดน้ำแข็งเป็น `admin` เท่านั้น; ผู้ใช้ที่ไม่พบโปรไฟล์หรือถูกพักใช้งานต้องเข้าใช้งานไม่ได้

## สัญญาการทำงาน Phase 3

- รอบส่งใช้จัดกลุ่มผู้ปฏิบัติงานและรายการขาย ไม่ใช่คลังสต๊อก และปิดรอบได้โดยไม่บังคับให้ทุกร้านมีรายการขาย
- ยอดสั่งจากโรงงานเป็นยอดตั้งต้นหรือยอดเติมของวัน โดยถือว่าโรงงานส่งครบตามยอดสั่ง เพราะหัวหน้าไม่ได้ตรวจรับตอนขึ้นรถ ระบบจึงไม่สรุปว่าส่วนต่างเกิดจากโรงงาน
- stock ledger ต่อเนื่องทั้งวันและแยกยอดตามรถบรรทุก ทีมส่ง พนักงาน รถเล็ก จุดปฏิบัติงาน จุดหน้ารถ และถังสำรอง; `courier` รับน้ำแข็งจากรถเข้าจุดถือครอง active ชนิดทีมส่งหรือรถเล็กที่ผูกกับตนเพียงหนึ่งจุด แล้วรายการส่งจะ snapshot และหักจากจุดนั้น ส่วน `round_lead` และ `admin` ยังคงใช้จุดต้นทางที่ตั้งค่าไว้กับร้าน
- `record_delivery` และ `record_stock_movement` ใช้ lock ตามวันที่ให้บริการชุดเดียวกัน ป้องกันการตัดสต๊อกพร้อมกันเกินยอดและปฏิเสธยอดติดลบ
- การสั่งจากโรงงานผูกกับวันที่ให้บริการและรถบรรทุกที่เปิดใช้งาน จึงทำได้ก่อนเปิดรอบและสั่งเพิ่มได้ระหว่างวัน; หลังปิดรอบยังโอนกลับ บันทึกของเสีย และคืนโรงงานได้จนกว่าจะปิดวัน
- การตรวจนับเก็บ snapshot ยอดตามระบบ ยอดจริง ส่วนต่าง ผู้ตรวจ เวลา และหมายเหตุ โดยไม่แก้ ledger หรือเดาสาเหตุให้อัตโนมัติ
- การแก้รายการส่งต้องมีเหตุผล รายการเดิมถูกยกเลิกแทนการลบ สต๊อกเดิมถูกคืนก่อนบันทึกรายการทดแทน และ audit log เชื่อมทั้งสองรายการ
- การปิดวันยึด `service_date` เป็นหลัก: หากมีรอบต้องปิดทุกรอบก่อน แต่วันที่มีเฉพาะยอดสั่งโรงงานและไม่มีรอบก็ตรวจนับและปิดวันได้ โดยล็อกยอด เก็บ snapshot ปรับตามยอดนับจริง รวบรวมยอดจากจุดย่อย และบันทึกยอดที่เหลือส่งคืนโรงงานใน transaction เดียว
- หลังปิดวัน ระบบปฏิเสธรอบใหม่ รายการส่ง และ movement ของวันนั้น วันถัดไปเริ่มจากยอดสั่งโรงงานใหม่โดยไม่มีสต๊อกยกมา
- โครงสร้างตึก/โซน/ร้านและจุดถือครองยังใช้ RPC และ RLS จาก migrations `0001`–`0009`; ร้านที่เปิดใช้งานถูกเพิ่มในรอบใหม่อัตโนมัติ และร้านต้องอ้างโซนของตึกเดียวกัน
- migration `0009_phase_3_review_fixes.sql` บังคับให้แอดมินบันทึกชนิดน้ำแข็งผ่าน RPC และไม่ให้พักใช้ชนิดที่ยังมียอดในวันที่ยังไม่ปิด พร้อมแยกประวัติสั่งโรงงานให้กรองประเภทก่อนจำกัดจำนวนรายการ
- migration `0011_round_independent_factory_orders.sql` ทำให้ `round_id` เป็นข้อมูลอ้างอิงที่ไม่บังคับสำหรับคำสั่งโรงงาน snapshot และ daily close, เพิ่ม RPC ตามวันที่และรถ, เปลี่ยน stock summary/count/close เป็น date-first โดยยังรับ `round_id` เดิมได้, ล็อกยอดทั้งวันร่วมกับ daily close และคงประวัติ movement เดิมไว้โดยไม่ย้ายหรือเขียนทับข้อมูล
- migration `0015_employee_assigned_stock_flow.sql` เพิ่ม `get_employee_stock_state` และ `record_employee_stock_transfer` สำหรับ flow เลือกรอบและโอนสต๊อกจากรถเข้าจุดถือครองของพนักงานแบบ idempotent พร้อมปรับ `record_delivery` ให้รายการของ `courier` ใช้จุดถือครองนั้นเป็นแหล่งตัดสต๊อก

## นอกขอบเขต

- ราคา เงื่อนไขชำระ เครดิต สรุปยอด เก็บเงิน ลูกหนี้ ใบแจ้งยอด ใบเสร็จ และการพิมพ์เอกสารเป็น Phase 3.5
- Phase 3 ไม่เก็บราคาจำลองหรือสร้างข้อมูลการเงินชั่วคราวไว้ในหน้าสั่งโรงงานหรือ stock ledger

## การตรวจสอบล่าสุด

- `npm test` ครอบคลุม role-to-view contract, flow พนักงานเลือกรอบ รับสต๊อกเข้าจุดถือครอง และตัดรายการส่งจากจุดนั้น, retry/history/ยอดรวมจาก server, factory-order, stock locking, dashboard, admin settings, การแก้รายการส่ง และการปิดสต๊อกสิ้นวัน โดย regression ฝั่งฐานข้อมูลรันกับ PostgreSQL-compatible PGlite จริง
- `npm run build` ผ่าน; Vite ยังแจ้งเตือน JavaScript chunk เกิน `500 kB` แต่ไม่ใช่ build failure
- migrations จนถึง `0015_employee_assigned_stock_flow.sql` ยังไม่ได้ apply และทดสอบกับ Supabase จริงในรอบแก้นี้ เพราะ workspace มีเฉพาะ frontend anon credentials และไม่มี Supabase CLI/project link
- ชนิดน้ำแข็งมาตรฐานที่เปิดใช้งานอยู่มี 3 รายการ: ก้อน, โม่ และเล็ก
- ทดสอบ flow จริงด้วยรอบ `ACCEPTANCE-PHASE3-20260715` สำหรับวันที่บริการ `2026-07-14` ครบ: เปิดรอบ → สั่งโรงงาน → retry ด้วย idempotency key เดิม → โอนสต๊อก → บันทึกส่ง → retry → แก้ยอดพร้อมเหตุผล → ตรวจนับ → ปิดรอบ → บันทึกของเสียหลังปิดรอบ → ปิดวัน → retry การปิดวัน
- ผล acceptance ตรงตาม contract: factory order, delivery และ daily close มีอย่างละหนึ่งรายการแม้ส่งซ้ำ; มี revision หนึ่งรายการ; ร้านทั้งหมด 1 ร้านส่งสำเร็จ 1 ร้าน; หลังปิดวันยอดทุกจุดเป็นศูนย์ และระบบปฏิเสธการเปิดรอบใหม่ย้อนหลังในวันที่ปิดแล้ว
- ตรวจสิทธิ์จาก session จริงแล้ว: `admin` เห็น 9 เมนูและจัดการข้อมูลอ้างอิงได้, `round_lead` เห็น 6 เมนูและถูกปฏิเสธเมื่อเรียก `save_shop` หรือเพิ่มชนิดน้ำแข็งโดยตรง, ส่วน session ที่ปรับเป็น `courier` ชั่วคราวถูกปฏิเสธจาก RPC สรุปสต๊อกและตรวจรายการส่ง ก่อนคืนบทบาทเป็น `round_lead`
- ตรวจหน้าหลังล็อกอินด้วย browser ทั้ง desktop และ mobile `390 × 844` แล้ว หน้า dashboard, สั่งโรงงาน, ควบคุมรอบ, stock operations, stock settings และข้อมูลอ้างอิงของแอดมินโหลดข้อมูลจริงตามสิทธิ์ ไม่พบ horizontal overflow, console warning หรือ console error
- ข้อมูลรับงานข้างต้นถูกเก็บไว้ใน Supabase เป็นหลักฐานที่ระบุชื่อ `ACCEPTANCE` ชัดเจน รวมถึง audit log; ไม่ลบย้อนหลังเพื่อรักษา ledger และหลักฐานการทดสอบ
