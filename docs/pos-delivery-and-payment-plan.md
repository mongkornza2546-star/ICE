# แผนปรับหน้าบันทึกส่งน้ำแข็งเป็น POS พร้อมระบบการเงิน

สถานะ: **แผนพร้อมนำไปพัฒนา**

ขอบเขตเอกสาร: หน้าบันทึกส่งน้ำแข็ง ราคา การรับชำระ รอบเก็บเงิน และเครดิต โดยยังไม่รวมเอกสารบัญชีและงานพิมพ์

เอกสารนี้รวมแนวทางออกแบบหน้าบันทึกส่งน้ำแข็งแบบ POS กับตำแหน่งตั้งค่าราคาและการชำระ เพื่อให้ทีมพัฒนานำไปทำงานต่อได้จากข้อกำหนดชุดเดียว โดยอ้างอิง [ข้อกำหนดระบบ](requirements.md), [กฎการปฏิบัติงาน](phase-0/operating-rules.md), [Flow การส่งและรับชำระ](employee-delivery-payment-flow.md) และ [แผนสรุปยอดสิ้นวัน](daily-billing-and-credit-plan.md)

## 1. เป้าหมาย

หน้าหน้างานต้องใช้ง่ายเหมือน POS พนักงานเห็นเฉพาะสิ่งที่ต้องทำจริง:

1. เลือกร้าน
2. เลือกชนิดน้ำแข็ง
3. ใส่จำนวน
4. ตรวจรายการและเงื่อนไขชำระ
5. บันทึกส่ง

ระบบต้องเลือกหรือคำนวณข้อมูลต่อไปนี้ให้อัตโนมัติ:

- วันที่ธุรกิจจาก `service_date` ของรอบส่ง
- รอบส่ง เมื่อผู้ใช้มีรอบเปิดที่เลือกได้เพียงรอบเดียว
- จุดตัดสต๊อกตามบทบาทและการตั้งค่าร้าน
- ราคาต่อหน่วยจาก master data
- ประเภทลูกค้าและตัวเลือกการชำระจากข้อมูลร้าน
- ยอดต่อรายการ ยอดรวม ยอดค้าง และวงเงินเครดิต

พนักงานห้ามแก้ราคา จุดตัดสต๊อก หรือเงื่อนไขเครดิตจากหน้า POS

## 2. สภาพระบบปัจจุบัน

### 2.1 สิ่งที่รองรับแล้ว

- การส่งหนึ่งครั้งรองรับน้ำแข็งหลายชนิดผ่าน `delivery_event` หนึ่งรายการและ `delivery_items` หลายรายการ
- `record_delivery` ตรวจรอบ สิทธิ์ จำนวนบวก ชนิดน้ำแข็งที่เปิดใช้ และสต๊อกคงเหลือก่อนบันทึกแบบ transaction เดียว
- มี idempotency key เพื่อป้องกันการกดซ้ำหรือ retry แล้วเกิดรายการซ้ำ
- พนักงานส่งใช้จุดถือครองที่ผูกกับผู้ใช้ ส่วนหัวหน้าและแอดมินใช้จุดสต๊อกที่ผูกกับร้าน
- มีการค้นหาและกรองร้านตามตึก/โซน ประวัติวันนี้ สถานะร้าน และ Flow แจ้งเหตุส่งไม่ได้
- Frontend หลักอยู่ที่ [EmployeeDeliveryWorkspace](../src/EmployeeDeliveryWorkspace.tsx) และใช้ร่วมกันหลายบทบาท

### 2.2 สิ่งที่ยังไม่มี

- ราคากลาง ราคาพิเศษรายร้าน และช่วงวันที่ราคาเริ่มมีผล
- Price snapshot ในรายการส่ง
- `payment_term`, `payment_method` และสถานะชำระที่คำนวณจากยอดเงินจริง
- รายการเรียกเก็บ การรับเงิน การจัดสรรเงิน ยอดค้าง และลูกหนี้เครดิต
- รอบเก็บเงินท้ายวันที่แยกจากรอบส่ง
- Flow ขออนุมัติเมื่อร้านไม่อนุญาตยอดค้างหรือเครดิตเกินวงเงิน

`shops.payment_status` เดิม (`unknown`, `paid`, `unpaid`) เป็นเพียงธงสถานะเก่า ไม่ใช่เงื่อนไขว่าจะให้ร้าน `จ่ายทันที`, `เก็บท้ายวัน` หรือ `เครดิต` และห้ามนำมาแปลงเป็น payment term อัตโนมัติ

## 3. ข้อตกลงหลักของแผน

| เรื่อง | ข้อสรุป |
|---|---|
| ผู้ใช้งาน | ใช้ POS เดียวกันสำหรับพนักงาน หัวหน้า และแอดมิน |
| จำนวน | จำนวนเต็มตามหน่วยของชนิดน้ำแข็ง เช่น ถุงหรือแถว |
| ราคา | ราคากลางรายชนิด + ราคาพิเศษรายร้าน |
| Payment term | ร้านกำหนดตัวเลือกที่อนุญาตและค่าเริ่มต้น |
| เครดิต | เป็นเงื่อนไขที่ล็อกจากร้าน พนักงานเปลี่ยนเองไม่ได้ |
| จ่ายทันที | บันทึกส่งและตัดสต๊อกก่อน แล้วจึงบันทึกรับเงินจริง |
| เก็บท้ายวัน | ใช้รอบเก็บเงินแยกจากรอบส่งและมอบหมายพนักงานได้ |
| ยอดค้าง/เกินวงเงิน | ใช้นโยบายรายร้านและคำขออนุมัติจากหัวหน้าหรือแอดมิน |
| ช่องทางรับเงิน | เงินสด โอน และ QR โดยบันทึกหลักฐาน แต่ยังไม่เชื่อมธนาคารหรือ Dynamic QR |
| เอกสาร | ยังไม่ทำเลขเอกสาร ใบสรุปยอด ใบแจ้งยอด ใบเสร็จ หรือการพิมพ์ |

## 4. หน้าบันทึกส่งแบบ POS

### 4.1 Desktop และ Tablet

ใช้หน้าจอเดียวสามส่วน โดยอิงพื้นที่จริงของ workspace ไม่ใช่ความกว้าง viewport อย่างเดียว:

| ส่วน | เนื้อหา |
|---|---|
| ซ้าย — เลือกร้าน | ค้นหารหัส/ชื่อร้าน, ตัวกรองตึกและโซน, รายการร้านแบบแถวสั้น, สถานะวันนี้ และ active state ของร้านที่เลือก |
| กลาง — เลือกสินค้า | ปุ่มสินค้าแบบ POS แสดงชื่อ หน่วย ราคา และจำนวนที่อยู่ในตะกร้า |
| ขวา — จำนวนและสรุป | สินค้าที่กำลังแก้, keypad, ตะกร้า, ยอดรวม, เงื่อนไขชำระ, ล้างรายการ และบันทึกส่ง |

รายการร้านต้องเปลี่ยนจากการ์ดรูปขนาดใหญ่เป็นแถวแบบกระชับ เพื่อให้ค้นหาและเลือกจากร้านจำนวนมากได้เร็ว รูปร้านยังแสดงได้เป็น thumbnail แต่ไม่ใช่องค์ประกอบหลัก

### 4.2 Mobile และจอเล็ก

เมื่อพื้นที่ workspace ต่ำกว่า 760px ให้เปลี่ยนเป็นสามขั้น:

1. **เลือกร้าน** — ค้นหา กรอง และเลือกร้าน
2. **เลือกรายการ** — เลือกสินค้า ใส่จำนวน และเพิ่มได้หลายชนิด
3. **ตรวจและบันทึก** — ตรวจร้าน ตะกร้า ราคา ยอดรวม และเงื่อนไขชำระ

การย้อนกลับระหว่างขั้นต้องรักษา draft เดิมไว้ การเปลี่ยนร้านขณะมีสินค้าในตะกร้าต้องขอยืนยันก่อนล้างรายการ

### 4.3 ปุ่มสินค้าและ Keypad

ปุ่มสินค้าต้อง:

- สูงและกว้างพอสำหรับการสัมผัสอย่างน้อย 48×48px
- แสดงชื่อเต็ม หน่วย ราคา และจำนวนปัจจุบัน
- ใช้ `aria-pressed` และ active state ที่มีทั้งสี เส้นขอบ และเครื่องหมาย ไม่พึ่งสีเพียงอย่างเดียว
- ใช้รูปชนิดน้ำแข็งจริงเมื่อมี และใช้ icon library เดิมเป็น fallback

Keypad เรียงแบบ POS:

```text
1  2  3
4  5  6
7  8  9
ล้าง  0  ⌫
```

กติกาการกรอก:

- ตัวเลขต่อหลัก เช่น กด `1` แล้ว `2` ได้ `12`
- `⌫` ลบหนึ่งหลัก และ `ล้าง` ตั้งเป็นศูนย์
- จำนวนศูนย์นำสินค้านั้นออกจากตะกร้า
- เปลี่ยนสินค้าแล้วจำนวนของสินค้าอื่นต้องไม่หาย
- ห้ามกรอกเกินสต๊อกต้นทางที่ server คืนมาใน POS context
- ปุ่มเลขบนแป้นพิมพ์จริงและ Numpad ทำงานเมื่อ focus ไม่ได้อยู่ในช่องค้นหา

### 4.4 ตะกร้าและปุ่มบันทึก

ตะกร้าแสดง:

- ร้าน รหัสร้าน ตึกและโซน
- น้ำแข็งแต่ละชนิด จำนวน หน่วย ราคาต่อหน่วย และยอดต่อรายการ
- ยอดรวม
- เงื่อนไขชำระและสถานะเครดิตที่เกี่ยวข้อง
- ปุ่มลบรายบรรทัด, **ล้างรายการ** และ **บันทึกส่ง**

ปุ่มบันทึกเปิดใช้งานเมื่อมีร้าน มีสินค้าอย่างน้อยหนึ่งรายการ ข้อมูลราคาครบ เงื่อนไขชำระถูกต้อง และรอบยังเปิดอยู่เท่านั้น

Flow “แจ้งเหตุส่งไม่ได้” และประวัติวันนี้ยังคงอยู่ แต่ย้ายเป็นการทำงานรอง ไม่แย่งพื้นที่จากงานบันทึกส่งปกติ

## 5. เงื่อนไขการชำระ

ระบบแยกข้อมูลสามมิติออกจากกัน:

| ข้อมูล | ความหมาย | ค่า |
|---|---|---|
| `payment_term` | ร้านกำหนดว่าจะจ่ายเมื่อใด | `immediate`, `end_of_day`, `credit` |
| `payment_method` | ช่องทางที่รับเงินจริง | `cash`, `bank_transfer`, `qr` |
| สถานะชำระ | ผลจากยอดเรียกเก็บลบยอดรับที่จัดสรรแล้ว | `unpaid`, `partial`, `paid` |

สถานะชำระต้องคำนวณจากข้อมูลการเงิน ห้ามให้พนักงานเลือกสถานะเอง

### 5.1 จ่ายทันที

1. POS ตรวจราคา สต๊อก และเงื่อนไขร้าน
2. `record_delivery` สร้างรายการส่ง สร้างยอดเรียกเก็บ snapshot ราคา และตัดสต๊อกใน transaction เดียว
3. เมื่อรายการส่งสำเร็จ จึงเปิด payment sheet
4. พนักงานเลือกเฉพาะวิธีชำระที่ร้านอนุญาต ใส่ยอดรับ และแนบเลขอ้างอิงหรือรูปหลักฐานตามการตั้งค่า
5. เงินสดรองรับเงินทอน ส่วนโอน/QR ห้ามบันทึกเกินยอดที่ต้องจัดสรร
6. หากรับไม่ครบ ระบบคงยอดที่เหลือเป็นยอดค้างและตรวจนโยบายร้าน

หากขั้นรับเงินล้มเหลว รายการส่งต้องยังอยู่และแสดงเป็นยอดยังไม่ได้รับชำระ ห้ามย้อนการตัดสต๊อกอัตโนมัติ

### 5.2 เก็บท้ายวัน

“เก็บท้ายวัน” เป็นงานแยกจากการส่งน้ำแข็งรอบสุดท้าย:

- หัวหน้าเปิดรอบเก็บเงินตาม `service_date` และมอบหมายพนักงานผู้เก็บ
- คิวรวมร้านที่มีรายการ `end_of_day` ค้างอยู่จากทุกครั้งที่ส่งในวันนั้น
- ร้านต้องอยู่ในคิวแม้ไม่ได้รับน้ำแข็งในรอบส่งสุดท้าย
- พนักงานหนึ่งคนสามารถส่งน้ำแข็ง ขณะที่อีกคนเดินเก็บเงินได้
- ก่อนบันทึกรับเงิน server ต้องคำนวณยอดค้างล่าสุดอีกครั้งเพื่อป้องกันยอด stale
- หากมีรายการส่งเพิ่มหลังเก็บเงินแล้ว ร้านต้องกลับมาแสดงสถานะ **มียอดเพิ่ม**
- หากร้านปิดหรือจ่ายไม่ครบ ยอดที่เหลือคงเป็นยอดค้างโดยไม่แก้ยอดขายเดิม

การบันทึกรับเงินต้อง serialize ตามร้านและวันที่ พร้อม expected outstanding amount และ idempotency key เพื่อป้องกันพนักงานสองคนบันทึกยอดเดียวกัน

### 5.3 เครดิต

- ร้านเครดิตถูกล็อกจากข้อมูลร้านและไม่แสดงตัวเลือก immediate/end-of-day ให้พนักงาน
- รองรับกำหนดวันเครดิตหรือครบกำหนดสิ้นเดือน พร้อมวงเงินเครดิตที่เป็นค่าว่างได้เมื่อไม่จำกัด
- ก่อนบันทึกส่ง server คำนวณยอดใช้วงเงินรวมรายการใหม่
- หากเกินวงเงิน ต้องมี approval ที่อนุมัติแล้วและตรงกับร้าน รายการสินค้า ยอดเงิน และ payment term เดียวกัน
- รายการเครดิตเข้าสู่หน้าลูกหนี้พร้อมยอดตั้งต้น ยอดรับ ยอดค้าง และวันครบกำหนด

### 5.4 คำขออนุมัติ

พนักงานส่งคำขอจาก POS ได้สองกรณี:

- ร้านไม่อนุญาตยอดค้าง แต่กำลังจะรับเงินไม่ครบ
- ร้านเครดิตจะเกินวงเงิน

คำขอต้องมี payload fingerprint, เหตุผล, ผู้ขอ, ผู้อนุมัติ, เวลา และสถานะ `pending/approved/rejected/used` การอนุมัติใช้ได้ครั้งเดียวกับ payload เดิมและหมดอายุเมื่อวันธุรกิจหรือรอบงานที่เกี่ยวข้องสิ้นสุด

## 6. ตำแหน่งตั้งค่าราคาและการชำระ

### 6.1 ราคากลางใน AdminReferenceSettings

เพิ่มส่วน **ชนิดน้ำแข็งและราคากลาง** ใน [AdminReferenceSettings](../src/AdminReferenceSettings.tsx)

ต่อหนึ่งชนิดน้ำแข็งให้ตั้งค่า:

- ชื่อและหน่วยขาย
- ราคากลาง
- วันที่เริ่มใช้และวันที่สิ้นสุดแบบไม่บังคับ
- สถานะใช้งาน
- ประวัติราคาแบบอ่านอย่างเดียว

การเพิ่มราคาใหม่ต้องสร้างแถวช่วงราคาใหม่ ห้ามแก้ทับราคาที่เคยถูกใช้แล้ว และห้ามมีช่วงราคากลางซ้อนกันสำหรับชนิดเดียวกัน

### 6.2 เงื่อนไขชำระใน ShopSettings

เพิ่มส่วน **การชำระเงิน** ใน [ShopSettings](../src/ShopSettings.tsx) เพราะแต่ละร้านมีเงื่อนไขต่างกัน:

- Payment term เริ่มต้น
- อนุญาต `จ่ายทันที` หรือ `เก็บท้ายวัน` อย่างใดอย่างหนึ่งหรือทั้งสองอย่าง
- ร้านเครดิต ซึ่งเป็นโหมดเฉพาะและล็อกตัวเลือกอื่น
- วิธีชำระที่อนุญาต: เงินสด โอน และ QR
- วิธีชำระเริ่มต้น
- ข้อกำหนดเลขอ้างอิงหรือรูปหลักฐานรายวิธี
- อนุญาตยอดค้างหรือไม่
- รูปแบบเครดิต จำนวนวันเครดิต/สิ้นเดือน และวงเงินเครดิต

ค่าเริ่มต้นต้องเป็นหนึ่งในตัวเลือกที่อนุญาต ร้านเครดิตต้องไม่เปิด immediate/end-of-day พร้อมกัน

### 6.3 ราคาพิเศษใน ShopSettings

เพิ่มส่วน **ราคาน้ำแข็งของร้าน** ในหน้าร้านเดียวกัน โดยแสดงตาราง:

| ชนิดน้ำแข็ง | หน่วย | ราคากลางปัจจุบัน | ราคาพิเศษ | เริ่มใช้ | สิ้นสุด |
|---|---|---:|---:|---|---|

ราคาที่ POS ใช้เรียงลำดับดังนี้:

1. ราคาพิเศษของร้านและชนิดน้ำแข็งที่มีผลใน `service_date`
2. ราคากลางของชนิดน้ำแข็งที่มีผลใน `service_date`
3. หากไม่พบราคา ให้บล็อกการส่งและแจ้งชื่อร้าน/ชนิดน้ำแข็งที่ต้องตั้งค่า

จำนวนเงินใช้ `numeric(12,2)` และราคาที่ resolve แล้วต้องถูก snapshot ตอนบันทึกส่ง

### 6.4 Bulk setup และ Readiness

เนื่องจากมีร้านจำนวนมาก ต้องมีเครื่องมือหลังบ้านสำหรับ:

- กำหนด payment profile ให้ร้านหลายร้านพร้อมกัน
- ดูร้านที่ยังไม่มี payment profile
- ดูชนิดน้ำแข็งที่ไม่มีราคากลางในวันที่ใช้งาน
- ดูร้านที่มีราคาพิเศษหมดอายุหรือช่วงราคาซ้อนกัน
- ยืนยันว่า active shops และ active ice types พร้อมก่อนเปิด POS การเงิน

## 7. Interfaces และ API

### 7.1 TypeScript types

เพิ่ม public types อย่างน้อย:

```ts
type PaymentTerm = 'immediate' | 'end_of_day' | 'credit';
type PaymentMethod = 'cash' | 'bank_transfer' | 'qr';
type FinancialPaymentStatus = 'unpaid' | 'partial' | 'paid';
```

`DeliveryPosContext` ต้องมีข้อมูล:

- รอบและ `service_date`
- ร้านและ `round_stop_id`
- จุดตัดสต๊อกที่ resolve ตามบทบาท
- สินค้าที่เปิดใช้ พร้อมหน่วย รูป ยอดคงเหลือ ราคาที่ resolve และแหล่งราคา
- allowed/default payment terms และ payment methods
- นโยบายยอดค้าง วันเครดิต วงเงิน และวงเงินคงเหลือ

Payload บันทึกส่งเพิ่ม `paymentTerm` และ `approvalId` แบบ optional ส่วนผลลัพธ์ต้องคืน `deliveryEventId`, `chargeId`, รายการราคาที่ snapshot, `totalAmount`, payment term, payment status และ due date เมื่อเป็นเครดิต

### 7.2 RPC ที่ต้องเพิ่มหรือขยาย

- `get_delivery_pos_context(round_stop_id)` — โหลดข้อมูลที่เชื่อถือได้หลังเลือกร้าน
- `record_delivery(...)` — ขยายให้ resolve ราคา ตรวจ payment term สร้าง charge และ snapshot การเงินใน transaction เดียวกับการตัดสต๊อก
- `record_payment(...)` — สร้างการรับเงินและจัดสรรยอดแบบ idempotent
- `void_payment(...)` — ยกเลิกรับเงินโดยเก็บเหตุผลและ audit log
- `open_collection_run(...)`, `get_collection_run_queue(...)`, `close_collection_run(...)`
- `request_financial_approval(...)`, `decide_financial_approval(...)`
- `get_credit_receivables(...)`

Client ส่งเพียงตัวเลือกและจำนวนที่ผู้ใช้กรอก ห้ามส่งราคาที่ถือเป็นแหล่งจริง Server ต้อง resolve ราคา สต๊อก สิทธิ์ และยอดคงค้างใหม่ทุกครั้งก่อนเขียนข้อมูล

Idempotency ของรายการใหม่ต้องเก็บ request fingerprint และปฏิเสธกรณี key เดิมแต่ร้าน สินค้า จำนวน payment term หรือยอดรับต่างจากคำขอเดิม

## 8. โครงสร้างข้อมูลเป้าหมาย

เพิ่มข้อมูลเป็นกลุ่มดังนี้:

### ราคาและร้าน

- ราคากลางรายชนิดน้ำแข็งพร้อมช่วงวันที่มีผล
- ราคาพิเศษรายร้าน/ชนิดพร้อมช่วงวันที่มีผล
- Payment profile หนึ่งชุดต่อร้าน

### รายการส่งและยอดเรียกเก็บ

- `delivery_items` เก็บ `unit_price`, `line_total` และ price source snapshot
- รายการ charge แบบหนึ่งต่อหนึ่งกับ delivery event ที่เป็นการส่งสำเร็จ
- Charge เก็บร้าน วันที่ธุรกิจ payment term ยอดตั้งต้น due date และสถานะใช้งาน

### การรับเงิน

- Payments เก็บวิธี ยอดรับ ยอดจัดสรร เงินทอน เลขอ้างอิง หลักฐาน ผู้รับ เวลา และ idempotency key
- Payment allocations รองรับเงินหนึ่งรายการตัดหลาย charges
- สถานะชำระคำนวณจากยอด charge ลบ allocations ที่ยังใช้งาน

### รอบเก็บเงินและการอนุมัติ

- Collection runs แยกจาก delivery rounds และมีสมาชิกผู้รับผิดชอบ
- Queue ร้านคำนวณจาก charge คงค้างของวันที่กำหนด และรองรับยอดเพิ่มระหว่างรอบ
- Approval requests ผูกกับ payload fingerprint และใช้ได้ครั้งเดียว

ทุกตารางใหม่ต้องมี RLS และสิทธิ์สอดคล้องกับบทบาท: courier เห็นเฉพาะรอบส่ง/รอบเก็บเงินที่ได้รับมอบหมายและคำขอของตน ส่วน round lead/admin จัดการรอบ การอนุมัติ และลูกหนี้ได้

## 9. การแก้ไข ยกเลิก และข้อมูลเดิม

- รายการส่งที่ยังไม่มีการจัดสรรเงินสามารถยกเลิก charge พร้อม delivery event ได้ใน transaction เดียว
- รายการที่จ่ายแล้วหรือจ่ายบางส่วนต้อง void การรับเงินพร้อมเหตุผลก่อนแก้หรือยกเลิก ห้ามลบ payment หรือ allocation
- การแก้จำนวนต้องใช้ราคาที่มีผลใน `service_date` เดิมและสร้างประวัติ correction ตามกลไกปัจจุบัน
- รายการส่งก่อนเปิดระบบการเงินให้เป็น **legacy unpriced** โดยไม่เดาหรือ backfill ราคา
- Legacy rows ไม่เข้าสู่คิวเก็บเงินหรือลูกหนี้ใหม่ เว้นแต่มีขั้นตอน migration ที่ผู้ดูแลตรวจและยืนยันเองในอนาคต
- คง `shops.payment_status` ไว้ชั่วคราวเพื่อ compatibility แต่หยุดใช้ใน POS ใหม่และรายงานการเงินใหม่

## 10. ลำดับการพัฒนาและเปิดใช้งาน

1. เพิ่ม schema ราคา Payment profile charges payments collections approvals และ RLS แบบ additive
2. เพิ่ม RPC และ integration tests โดยยังไม่เปลี่ยนหน้าหน้างาน
3. เพิ่มหน้าราคากลาง การชำระรายร้าน ราคาพิเศษ Bulk setup และ readiness report
4. ตั้งราคากลางและ payment profile ให้ active shops ครบ
5. ปรับ EmployeeDeliveryWorkspace เป็น POS และเปิดใช้ record-delivery contract ใหม่
6. เปิด Flow จ่ายทันที รอบเก็บเงิน คิวลูกหนี้ และคำขออนุมัติ
7. ตรวจข้อมูลจริงและ audit logs ก่อนหยุดใช้ legacy payment flag ในหน้าจอเดิม

แต่ละขั้นต้อง deploy ได้โดยไม่บังคับเดาราคาย้อนหลังหรือทำให้รายการส่งเดิมเปลี่ยนยอด

## 11. Test Plan

### 11.1 POS และตะกร้า

- เริ่มต้นเลือกได้เฉพาะร้าน สินค้าและปุ่มบันทึกยัง disabled
- เลือกร้านแล้วโหลดสินค้า ราคา หน่วย จุดสต๊อก และ payment profile ถูกต้อง
- Keypad `1,2` ได้ `12`, backspace เหลือ `1`, ล้างเป็นศูนย์
- ตะกร้ารองรับหลายชนิดและส่งเป็น delivery event เดียว
- เปลี่ยนร้านขณะมี draft ต้องยืนยันและยกเลิกแล้วข้อมูลเดิมไม่หาย
- Retry ใช้ idempotency key เดิมและไม่สร้างรายการซ้ำ

### 11.2 ราคาและการเงิน

- ราคาพิเศษมีลำดับเหนือราคากลางและเลือกตาม `service_date`
- การเปลี่ยนราคาใหม่ไม่เปลี่ยน delivery item เดิม
- ร้านไม่มีราคา/Payment profile ต้องบันทึกไม่ได้โดยไม่ตัดสต๊อก
- จ่ายเต็ม บางส่วน เงินสดเกินยอด และ payment failure หลังส่งทำงานตามกติกา
- เก็บท้ายวันรวมทุกรอบและรวมร้านที่ไม่รับน้ำแข็งรอบสุดท้าย
- รายการส่งเพิ่มระหว่างเก็บเงินทำให้ยอด refresh หรือร้านกลับเป็นมียอดเพิ่ม
- เครดิตคำนวณ due date วงเงิน และ approval ถูกต้อง
- Concurrent submit/collection และ retry ไม่สร้าง delivery, payment หรือ allocation ซ้ำ

### 11.3 สิทธิ์และประวัติ

- Courier ใช้จุดถือครองของตนและเห็นเฉพาะรอบที่ได้รับมอบหมาย
- Round lead/admin ใช้จุดสต๊อกของร้านและจัดการ collection/approval ได้
- การแก้ ยกเลิก void และอนุมัติทุกครั้งมี audit log
- Legacy unpriced ไม่เข้าสรุปยอดใหม่

### 11.4 Responsive และ Accessibility

- Desktop 1440×900 แสดงสามส่วนโดยไม่มี horizontal overflow
- Tablet 1024×768 และ 768×1024 ยังคง layout POS แบบกระชับ
- Mobile 390×844 และ 320×568 ใช้สามขั้นและ sticky CTA ไม่บังเนื้อหา
- ปุ่มสัมผัสอย่างน้อย 48px, contrast ข้อความอย่างน้อย 4.5:1 และ focus ring ชัด
- Tab/Shift+Tab, Enter/Space, keyboard digits และ Numpad ทำงานครบ
- Screen reader รับรู้ร้าน สินค้า หน่วย จำนวน ตะกร้า Payment term และผลการบันทึก

## 12. เกณฑ์รับงาน

- พนักงานเลือกร้าน เพิ่มน้ำแข็งหลายชนิดด้วย keypad และบันทึกครั้งเดียวได้
- วันที่ รอบ จุดตัดสต๊อก ราคา และประเภทลูกค้าถูกเลือกอัตโนมัติเมื่อข้อมูลไม่กำกวม
- Desktop/Tablet ใช้หน้าจอเดียว และ Mobile ใช้สามขั้นโดย draft ไม่หาย
- ราคากลางตั้งใน AdminReferenceSettings ส่วนเงื่อนไขชำระและราคาพิเศษตั้งใน ShopSettings
- รายการส่งเก็บ price snapshot และการเปลี่ยนราคาไม่กระทบยอดย้อนหลัง
- จ่ายทันที เก็บท้ายวัน เครดิต ยอดค้าง วงเงิน และ approval ทำงานตามกติกา
- รอบเก็บเงินทำงานแยกจากรอบส่งและรองรับพนักงานส่งกับพนักงานเก็บเงินทำงานพร้อมกัน
- ไม่มีการสร้างข้อมูลซ้ำจาก double-click, network retry หรือผู้เก็บเงินหลายคน
- ข้อมูล legacy ไม่ถูกแปลงหรือรวมยอดโดยการคาดเดา

## 13. ข้อสมมติและสิ่งที่ยังไม่รวม

- ราคาเป็นเงินบาทสุทธิ ยังไม่แยก VAT หรือส่วนลด
- จำนวนขายรอบนี้เป็นจำนวนเต็มและมีหนึ่งหน่วยต่อชนิดน้ำแข็ง
- การรองรับครึ่งถุงและการแยกจำนวนขายออกจากจำนวนตัดสต๊อกเป็นงานถัดไป และแผนนี้ไม่ยกเลิกข้อกำหนดระยะยาวดังกล่าว
- รองรับการบันทึกเงินสด โอน และ QR แต่ยังไม่สร้าง Dynamic QR หรือตรวจสถานะจากธนาคาร
- ยังไม่ทำเลขเอกสาร ใบสรุปยอด ใบแจ้งยอด ใบเสร็จ เครื่องพิมพ์ หรือข้อกำหนดภาษี
- การแก้ไขครั้งนี้ต้องรักษา Flow สต๊อก idempotency การแจ้งเหตุส่งไม่ได้ และประวัติวันนี้ที่มีอยู่แล้ว
