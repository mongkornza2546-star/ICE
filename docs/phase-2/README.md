# Phase 2: บัตรร้านและการบันทึกส่งสำหรับพนักงาน

สถานะ: flow แรกของหน้าพนักงานตาม contract ของ migration `0015_employee_assigned_stock_flow.sql` อยู่ใน workspace และ migrations จนถึง `0015` ยังรอ apply/ทดสอบกับ Supabase project จริง

## สิ่งที่เริ่มทำแล้ว

- migration `0002_phase_2_delivery_operations.sql` เพิ่ม RPC `create_delivery_round` ให้หัวหน้ารอบหรือแอดมินเปิดรอบแบบ atomic
- migration `0006_rounds_without_routes.sql` ปรับ RPC ให้เปิดรอบโดยสร้างสมาชิก คัดลอกร้านที่เปิดใช้งานทั้งหมดเป็น `round_stops` และสร้างแถวรองรับชนิดน้ำแข็งที่ยัง active โดยไม่ต้องกำหนดเส้นทาง; frontend ส่งค่า legacy นี้เป็นศูนย์เพราะยอดสต๊อกจริงเริ่มจากรายการสั่งโรงงานใน ledger รายวัน
- RPC `record_delivery` ให้พนักงานที่เป็นสมาชิกของรอบ (หรือหัวหน้ารอบ/แอดมิน) บันทึกน้ำแข็งหรือสถานะปัญหาแบบ atomic พร้อม idempotency key, audit log และอัปเดตสถานะบัตรร้าน โดยตอบข้อมูลรายการและรายการน้ำแข็งเดิมในการ retry
- migration `0015_employee_assigned_stock_flow.sql` เพิ่ม RPC อ่านยอดรถและจุดถือครองของพนักงาน, บันทึก stock transfer จากรถเข้าจุดที่ผูกกับพนักงานแบบ idempotent และปรับ `record_delivery` ให้ `courier` snapshot และหักสต๊อกจากจุดนั้น
- `courier` ต้องมีจุดถือครองที่เปิดใช้งานเพียงหนึ่งจุดชนิด `team` หรือ `small_vehicle`; ระบบปฏิเสธ flow หากไม่มีจุดหรือพบมากกว่าหนึ่งจุด ส่วน `round_lead` และ `admin` ยังคงใช้จุดต้นทางที่ตั้งค่าไว้กับร้าน
- RPC `get_round_shop_cards` คืนบัตรร้านเรียงตามอาคาร/โซน/รหัสร้าน กรองอาคารได้ และมีประวัติพร้อมยอดรวมของวันเดียวกันจากทุกรอบ
- migration `0003_phase_2_round_assignment_helpers.sql` เพิ่ม RPC `get_assignable_round_members` เพื่อให้หัวหน้ารอบ/แอดมินดึงรายชื่อผู้ใช้ active ไปเลือกผู้ร่วมรอบได้โดยไม่ต้องเปิด `public.users` ทั้งตารางผ่าน RLS
- frontend Vite + React ใน root ของ repo: `courier` ใช้หน้าจอเต็มแบบ POS ส่วนหัวหน้ารอบ/แอดมินใช้ workspace เดียวกันจากเมนูบันทึกส่งน้ำแข็ง
- หน้าพนักงานจัดลำดับให้เลือกรอบ รับน้ำแข็งจากรถเข้าสู่จุดถือครองที่ผูกกับตน แล้วจึงเลือกจำนวนจากจุดนั้นก่อนกรองตึก/โซนหรือค้นหารหัสและชื่อร้านปลายทาง
- เมื่อแตะร้าน ระบบแสดงหน้าตรวจสอบยอดกับร้านปลายทางและใช้ปุ่มยืนยันเพียงครั้งเดียว; บันทึกสำเร็จแล้วล้างยอดน้ำแข็งและกลับรายการร้านโดยเก็บคำค้นหา ตัวกรอง และตำแหน่งเลื่อนเดิม
- การ retry payload เดิมใช้ idempotency key และเวลาบันทึกเดิมแม้ออกจากร้านหรือ reload หน้า ป้องกัน response เก่าจากการเปลี่ยนรอบ และล็อกปุ่มย้อนกลับ/ยืนยันระหว่างรอผล
- ยอดและประวัติวันนี้รวมรายการของร้านจากทุกรอบในวันเดียวกัน และประวัติเหตุส่งไม่ได้แสดงสถานะกับหมายเหตุเดิม
- บัตรสถานะ `full_bin`, `closed_shop`, `no_access` และ `issue` ต้องมีหมายเหตุ; สถานะ `delivered` ต้องมีรายการน้ำแข็งอย่างน้อยหนึ่งประเภท

## ขั้นตอนหน้าพนักงานแบบ POS

1. เลือกรอบส่ง ถ้ามีรอบเปิดเพียงรอบเดียวระบบเลือกให้ แต่ถ้ามีหลายรอบพนักงานต้องเลือกเอง
2. ระบบโหลดรถบรรทุกและจุดถือครองชนิดทีมส่งหรือรถเล็กที่ผูกกับพนักงานเพียงหนึ่งจุด พร้อมยอดคงเหลือแยกตามชนิด
3. ใส่จำนวนที่รับจากรถและยืนยัน stock transfer เข้าจุดถือครอง; การรับเพิ่มสร้าง transfer ใหม่
4. เลือกจำนวนที่จะส่งจากจุดถือครอง ค้นหาหรือกรอง แล้วแตะบัตรร้านปลายทาง
5. ตรวจยอดกับชื่อร้าน แล้วกดยืนยันรายการส่งซึ่งจะ snapshot และหักสต๊อกจากจุดถือครอง
6. หากส่งไม่ได้ให้เปิดทางเลือกรอง เลือกเหตุและใส่หมายเหตุ; หากร้านไม่ซื้อในรอบนี้ให้กลับรายการร้านโดยไม่สร้างรายการ

## สิ่งที่ยังไม่ทำ

- ยังไม่ได้เชื่อม Supabase project จริงใน workspace นี้ จึงยังไม่ได้ทดสอบกับ Auth/RLS/RPC ของฐานข้อมูลจริง
- การแก้ไขรายการล่าสุด: ค่า edit window ยังไม่ได้อนุมัติใน Phase 0 จึงไม่เปิดสิทธิ์แก้ไขแก่พนักงาน
- ปิดรอบและตรวจยอดของหัวหน้า เป็นขอบเขต Phase 3
- ยังไม่มี offline queue, installable PWA หรือหน้าจอ retry สำหรับการ sync แบบ offline ซึ่งเป็นขอบเขต Phase 4
- ราคา การรับชำระ เครดิต ร้านหน้ารถ และการพิมพ์เอกสารยังไม่รวมในหน้าจอนี้

## สัญญาเรียกใช้งาน

เปิดรอบโดยเรียก `create_delivery_round(service_date, name, member_ids, loaded_quantities)` โดย `loaded_quantities` ยังอยู่เพื่อความเข้ากันได้ของ schema เดิม แต่ต้องส่งศูนย์ทุกชนิด เช่น:

```json
[{"ice_type_id":"UUID","quantity":0}]
```

บันทึกการส่งด้วย `record_delivery(round_stop_id, items, stop_status, note, client_recorded_at, idempotency_key)` โดย `items` เป็น JSON array รูปแบบเดียวกัน และ `idempotency_key` หนึ่งค่าใช้ต่อการกดยืนยันหนึ่งครั้งเท่านั้น

ตาม migration `0015_employee_assigned_stock_flow.sql` หน้าพนักงานอ่านยอดด้วย `get_employee_stock_state(round_id)` และรับน้ำแข็งด้วย `record_employee_stock_transfer(round_id, items, idempotency_key)`. RPC ทั้งสองเลือกจุดถือครอง active ชนิด `team` หรือ `small_vehicle` ที่ `assigned_user_id` ตรงกับผู้ใช้และต้องพบเพียงหนึ่งจุด. เมื่อ `courier` เรียก `record_delivery` ระบบ snapshot จุดนี้ไว้ใน `delivery_events.source_stock_location_id` และตรวจ/หักยอดจากจุดนี้ ส่วน `round_lead` และ `admin` ยังคงใช้ `shops.stock_location_id`

หลังรับสต๊อกเข้าจุดถือครองแล้ว frontend เรียก `get_round_shop_cards(round_id, building_id)` เพื่อแสดงบัตรร้านทั้งหมดในรอบและยอดรวมของวัน แทนการประกอบประวัติหลายรอบบนอุปกรณ์เอง พนักงานเป็นผู้เลือกร้านปลายทางที่จะไปส่งเอง

ห้ามใช้ `loaded_quantities` เป็นยอดสต๊อกหรือยอดสั่งโรงงาน สต๊อกตั้งต้นต้องมาจาก `record_factory_order(service_date, truck_location_id, ...)` โดยไม่ต้องเปิดรอบก่อน และต่อเนื่องข้ามทุกรอบในวันเดียวกัน

## วิธีรัน frontend scaffold

1. คัดลอก `.env.example` เป็น `.env.local`
2. ใส่ `VITE_SUPABASE_URL` และ `VITE_SUPABASE_ANON_KEY`
3. รัน `npm install`
4. รัน `npm run dev`

หน้าจอที่เริ่มใช้ได้ในรอบนี้:

- เข้าสู่ระบบด้วย Supabase Auth
- หัวหน้ารอบ/แอดมินเปิดรอบใหม่จาก RPC `create_delivery_round` โดยไม่ต้องเลือกเส้นทาง
- พนักงานเลือกรอบ รับน้ำแข็งจากรถเข้าสู่จุดถือครองที่ผูกกับตน แล้วเลือกบัตรร้านปลายทางและบันทึกรายการส่งผ่าน RPC `record_delivery`
- เห็นประวัติและยอดรวมของวันจาก RPC `get_round_shop_cards`
