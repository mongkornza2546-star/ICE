# แผนพัฒนาระบบบัตรร้านค้าดิจิทัล

เอกสารนี้แบ่งงานเป็น Phase ตามลำดับที่ช่วยให้เริ่มใช้งานหน้างานได้เร็ว และลดความเสี่ยงของข้อมูลยอดส่งก่อนเพิ่มฟังก์ชันขั้นสูง

## Phase 0 — ออกแบบและเตรียมข้อมูล (กำลังดำเนินการ)

**เป้าหมาย:** ยืนยันกติกาหน้างาน โครงสร้างสต๊อกเคลื่อนที่ และเตรียมข้อมูลร้านให้พร้อมก่อนเขียนระบบจริง

ขอบเขต:

- สรุปประเภทน้ำแข็งที่ใช้จริงและหน่วยนับ
- กำหนดหน่วยตัดสต๊อกจริงแยกจากหน่วยคิดเงิน เช่น เต็มถุงและครึ่งถุง
- นำเข้าข้อมูลร้าน อาคาร โซนย่อย ตำแหน่งอ้างอิง และข้อมูลติดต่อ โดยไม่บังคับลำดับเดินตายตัว
- กำหนดจุดถือครองสต๊อก ได้แก่ รถบรรทุก ทีมส่ง รถเล็ก จุดปฏิบัติงาน และถังสำรอง
- กำหนดบทบาทผู้ใช้: พนักงานส่ง หัวหน้ารอบ และแอดมิน
- ออกแบบข้อมูลคำสั่งซื้อจากโรงงาน โดยใช้ยอดสั่งเป็นสต๊อกตั้งต้น การโอนสต๊อก รายการส่ง สรุปยอด และ audit log
- ระบุข้อจำกัดว่าหัวหน้าไม่ได้ตรวจนับตอนโรงงานขึ้นของ จึงห้ามสรุปว่าส่วนต่างเกิดจากโรงงานโดยไม่มีหลักฐานการรับจริง
- ยืนยันราคาที่หัวหน้าหรือแอดมินตั้งแยกร้าน/ชนิด/รูปแบบการขาย
- ยืนยันเงื่อนไขจ่ายทันที เก็บท้ายวัน เครดิตเฉพาะร้าน รอบวางบิล และรูปแบบเลขที่เอกสาร
- ยืนยันวิธีนับน้ำแข็งที่แยกไว้สำหรับ รปภ. หรือบุคคลที่มารับจากรถ โดยไม่บังคับปริมาณรายคน
- ทำ wireframe หน้าพนักงานและหน้าควบคุมรอบบนโทรศัพท์
- กำหนดวิธีทำงานเมื่อไม่มีอินเทอร์เน็ตและกรณีซิงก์ซ้ำ

ผลส่งมอบ: ข้อมูลตั้งต้นร้านและโครงสร้างสถานที่, ผังหน้าจอ, ข้อกำหนดฐานข้อมูลสต๊อกและการขาย และกติกาการทำงานที่อนุมัติแล้ว

เอกสารทำงานของ Phase 0 อยู่ที่ [phase-0/README.md](phase-0/README.md) โดยต้องอนุมัติ
ชนิดน้ำแข็งและหน่วยนับ ข้อมูลร้านจริง ผู้รับผิดชอบแต่ละบทบาท และกติกาการแก้ไขก่อนเริ่ม Phase 1

## Phase 1 — ระบบพื้นฐานและจัดการข้อมูล (เริ่มพัฒนา)

**เป้าหมาย:** ให้แอดมินจัดการข้อมูลอ้างอิงและผู้ใช้ได้อย่างปลอดภัย

ขอบเขต:

- ระบบเข้าสู่ระบบและกำหนดสิทธิ์ตามบทบาท
- จัดการข้อมูลตามลำดับ `ตึก → โซนย่อยของตึก → ร้าน` โดยร้านต้องเลือกโซนย่อยที่มีอยู่จริง พร้อมตำแหน่งอ้างอิง รูปร้าน และสถานะร้าน
- จัดการประเภทน้ำแข็งและผู้ใช้งาน
- จัดการจุดถือครองสต๊อก รถบรรทุก รถเล็ก ทีมส่ง จุดปฏิบัติงาน และถังสำรอง
- จัดการถังประจำร้าน โดยแยกถังของร้านและถังที่เช่าจากผู้ขาย
- สร้างตารางข้อมูลรอบส่ง รายการส่ง และประวัติการแก้ไข
- สร้างบัญชีการเคลื่อนไหวสต๊อกสำหรับรายการที่ตรวจนับได้ โดยระบุต้นทาง ปลายทาง ชนิด จำนวน ผู้บันทึก และเวลา
- ให้รายการขายหักสต๊อกจากจุดสต๊อกย่อยที่ผูกกับร้านโดยอัตโนมัติ โดยไม่บังคับหัวหน้าบันทึกการหยิบของไปแต่ละร้านซ้ำ
- เตรียมโครงสร้างราคารายร้าน เงื่อนไขชำระและสิทธิ์เครดิตเฉพาะร้าน เอกสารเรียกเก็บ การรับชำระ และเลขที่เอกสารแบบไม่ซ้ำ
- บันทึกเหตุการณ์การแก้ไขและยกเลิกแบบไม่ลบข้อมูลจริง

เกณฑ์รับงาน: แอดมินเพิ่ม/แก้ไขร้าน ผู้ใช้ จุดถือครองสต๊อก และเงื่อนไขรายร้านได้ โดยผู้ใช้เข้าถึงได้เฉพาะสิ่งที่บทบาทอนุญาต

## Phase 2 — บัตรร้านและการบันทึกส่งสำหรับพนักงาน (กำลังพัฒนา)

**เป้าหมาย:** ให้พนักงานส่งบันทึกงานประจำวันได้จริงจากโทรศัพท์

ขอบเขต:

- รอบส่งเป็นกลุ่มรายการขาย/ผู้ปฏิบัติงานตามช่วงเวลา ไม่ใช่เส้นทางหรือคลังสต๊อก และไม่มีจำนวนมาตรฐานต่อวัน
- หัวหน้าดูรายการขายของแต่ละรอบได้ โดยไม่ต้องมอบหมายอาคาร โซน ร้าน หรือลำดับเดินให้พนักงาน
- ผูกรายการขายกับรอบส่ง ร้าน และจุดสต๊อกต้นทาง โดยร้านเดียวมีรายการได้หลายรอบในวันเดียวกัน
- หน้าพนักงานเลือกอาคาร โซน และบัตรร้านเอง พร้อมจัดลำดับการเดินหน้างานได้อิสระ
- ร้านอาจรับหรือไม่รับน้ำแข็งในแต่ละรอบตามยอดขายและน้ำแข็งที่เหลือจากรอบก่อน โดยไม่บังคับให้ทุกร้านมีรายการขายทุกรอบ
- ร้านที่ไม่รับน้ำแข็งในรอบนั้นไม่ต้องให้พนักงานกดสถานะหรือสร้างรายการใด ๆ
- บัตรร้านแสดงรูป รหัส ชื่อ ตำแหน่ง ประวัติการขาย ยอดรวมวันนี้ และสถานะจ่ายเงิน
- หน้าบันทึกส่งด้วยปุ่มเลือกประเภทน้ำแข็งและจำนวนขนาดใหญ่
- บันทึกจำนวนที่ใช้คิดเงินแยกจากจำนวนที่ตัดสต๊อกจริง สำหรับเต็มถุงและครึ่งถุง
- ไม่สร้างสถานะ `ยังมีน้ำแข็ง/ไม่รับรอบนี้` หรือ `ยังไม่ส่ง` ที่บังคับให้พนักงานกด ระบบดูว่ารอบใดมีการขายจากรายการขายจริงเท่านั้น
- ประวัติรายการหลายรอบในร้านเดียว และยอดรวมของวัน
- เก็บราคาต่อหน่วยและยอดเงิน snapshot กับรายการส่ง เพื่อคำนวณยอดสิ้นวันโดยไม่เปลี่ยนย้อนหลัง
- แก้ไขรายการล่าสุดตามสิทธิ์และช่วงเวลาที่กำหนด

เกณฑ์รับงาน: พนักงานเลือกลำดับเดินเองและกดเฉพาะเมื่อมีรายการขาย ร้านเดียวมีได้หลายรายการหรือไม่มีรายการในบางรอบ โดยไม่ต้องบันทึกสถานะเมื่อไม่ขาย และยอดรวมแสดงเฉพาะการขายจริงอย่างถูกต้อง

สถานะการพัฒนาและสัญญาเรียกใช้งานอยู่ที่ [phase-2/README.md](phase-2/README.md)

## Phase 3 — ควบคุมสต๊อกเคลื่อนที่ หน้าควบคุมรอบ และตรวจยอดหัวหน้า (implementation ครบใน workspace — รอทดสอบ Supabase จริง)

**เป้าหมาย:** ให้หัวหน้าควบคุมน้ำแข็งตั้งแต่บันทึกยอดสั่งโรงงานที่ใช้เป็นยอดตั้งต้น กระจายไปแต่ละจุด จนถึงตรวจยอดเมื่อจบรอบได้จากระบบ

ขอบเขต:

- แดชบอร์ดรอบ: จำนวนร้านที่มีรายการขาย ยอดขายของรอบ และปัญหาที่มีผู้แจ้ง โดยไม่บังคับติดตามสถานะร้านที่ไม่มีรายการขาย
- แสดงตำแหน่งงานล่าสุดของพนักงานจากรายการที่บันทึก (ไม่ใช่ GPS แบบติดตามสด)
- บันทึกจำนวนที่สั่งโรงงานแยกตามประเภท และใช้เป็นยอดสต๊อกรถตั้งต้นโดยสมมุติว่าโรงงานส่งครบ
- บังคับให้หัวหน้าระบุชนิดและจำนวนทุกครั้งที่สั่งจากโรงงาน รวมถึงการสั่งเพิ่มระหว่างวัน
- ไม่บันทึกยอดรับจริงจากโรงงาน เนื่องจากหัวหน้าไม่ได้อยู่ตรวจนับตอนขึ้นรถ
- รองรับการสั่งน้ำแข็งเพิ่มจากโรงงานระหว่างวันได้หลายครั้ง พร้อมเวลา ชนิด และจำนวน โดยเพิ่มยอดสั่งเข้าสต๊อกรถเมื่อของมาถึง
- ใช้รถบรรทุกเป็นคลังกลางเคลื่อนที่ และแสดงยอดคงเหลือปัจจุบันของรถ
- บันทึกการโอนน้ำแข็งจากรถไปจุด Skywalk, จุดตึก A, พนักงานตึก B แต่ละคน และรถเล็ก/พนักงานตึก C ตามยอดตรวจนับจริง
- บันทึกยอดส่งมอบให้พนักงาน/รถเล็กและยอดเหลือจริงเมื่อกลับมาแยกตามผู้รับและชนิด แล้วเทียบกับยอดคงเหลือตามระบบพร้อมส่วนต่างและหมายเหตุ
- ผูกการจ่ายของให้พนักงาน การรับเพิ่ม การคืนรถ การโอนเข้าถังสำรอง และการขายกับจุดถือครองจริง; รอบส่งใช้เป็นข้อมูลประกอบรายการขายเท่านั้น
- ให้หัวหน้าตรวจความคืบหน้ารายการขายของแต่ละรอบได้ โดยสต๊อกประจำวันต่อเนื่องตลอดเส้นทางและไม่ถูกปิดพร้อมรอบ
- รองรับอาคารที่มีหลายโซนย่อย และแบ่งงาน/สต๊อกตามทีมและจุดหมายจริง
- บันทึกการรับเพิ่ม คืนรถ เก็บในถังสำรอง และเสียหาย โดยไม่เขียนทับรายการโอนก่อนหน้า
- ให้รถเล็กกลับมารับน้ำแข็งเพิ่ม แล้วนำไปเติมตึก C, Skywalk หรือตึก A เป็นรายการโอนใหม่ได้หลายครั้ง
- แสดงยอดคงเหลือแยกตามรถ ทีม รถเล็ก จุดปฏิบัติงาน และถังสำรอง
- คำนวณยอดใช้ของจุดสต๊อกย่อย เช่น ตึก A จากรายการขายของร้านที่ผูกกับจุดนั้น แทนการให้หัวหน้าบันทึกการหยิบออกทุกครั้ง
- แยกน้ำแข็งสำหรับ รปภ. หรือบุคคลที่มารับจากรถออกเป็นสต๊อกย่อยครั้งเดียว
- บันทึกรายคนเพียงเหตุการณ์ว่ามีผู้มารับ ชนิดน้ำแข็ง และจ่ายหรือไม่จ่าย โดยไม่บังคับปริมาณรายคนและไม่ตัดสต๊อกรถซ้ำ
- ปิดยอดสต๊อกที่แยกไว้จากยอดตั้งต้น ยอดเหลือจริง และยอดเสีย/ละลายตอนจบ
- แสดงยอดคงเหลือตามระบบสิ้นวันแยกตามชนิดและจุดถือครอง พร้อมยอดรวมทั้งระบบ
- ให้หัวหน้ากรอกยอดนับจริงสิ้นวันแยกตามชนิดและจุดที่นับ
- เก็บยอดตามระบบ ยอดนับจริง ส่วนต่าง ผู้ตรวจนับ เวลา และหมายเหตุเป็น snapshot โดยไม่เขียนทับกัน
- บันทึกน้ำแข็งที่เหลือทั้งหมดว่าส่งกลับโรงงานหลังตรวจนับ และปิดสต๊อกของรถ/จุดสต๊อกย่อยเป็นศูนย์
- ไม่ยกสต๊อกน้ำแข็งข้ามวัน โดยวันใหม่เริ่มจากยอดสั่งโรงงานของวันนั้น
- ตรวจนับจริงและส่งน้ำแข็งที่เหลือกลับโรงงานหลังจบทุกรอบของวันเท่านั้น
- คำนวณยอดคงเหลือที่ควรมีของแต่ละจุดเทียบกับยอดนับจริง และแสดงส่วนต่างโดยไม่เดาสาเหตุ
- ปิดรอบได้โดยไม่บังคับว่าทุกร้านต้องมีรายการขาย และหากมียอดต่างหรือปัญหาที่มีผู้แจ้งให้เก็บไว้ในสรุปรอบ
- หัวหน้าแก้ไขรายการของทุกคนพร้อมบันทึกเหตุผล

เกณฑ์รับงาน: หัวหน้าตรวจย้อนหลังได้ว่าน้ำแข็งแต่ละชนิดถูกสั่งครั้งแรกและสั่งเพิ่มเท่าไร ถูกจ่ายไปจุดใด ใช้ไปจากยอดขายร้านเท่าไร เหลือตามระบบเท่าไร นับจริงสิ้นวันได้เท่าไร ส่งกลับโรงงานเท่าไร และพบส่วนต่างที่จุดใด โดยวันใหม่ไม่มีสต๊อกยกมาและระบบไม่กล่าวโทษโรงงานจากยอดสั่งที่ไม่ได้ตรวจนับรับจริง

สถานะ implementation ของส่วนหัวหน้าและลำดับงานที่เหลืออยู่ที่ [phase-3/README.md](phase-3/README.md)

## Phase 3.5 — สรุปยอดสิ้นวัน เก็บเงิน และพิมพ์เอกสาร

**เป้าหมาย:** ให้พนักงานรู้ว่าร้านใดจ่ายทันที ร้านใดเก็บท้ายวัน และให้เฉพาะร้านที่ได้รับสิทธิ์เครดิตสะสมยอดค้างได้ถูกต้อง

ขอบเขต:

- ให้หัวหน้าหรือแอดมินตั้งราคาแยกร้าน ชนิดน้ำแข็ง และรูปแบบการขาย พร้อมวันที่เริ่มมีผล
- เก็บราคาและยอดเงิน snapshot โดยแยกจำนวนคิดเงินออกจากจำนวนตัดสต๊อกจริง
- กำหนดเงื่อนไขรายร้านเป็น `จ่ายทันที`, `เก็บท้ายวัน`, `เครดิตตามจำนวนวัน` หรือ `วางบิลสิ้นเดือน`
- เครดิตต้องเปิดให้เฉพาะร้าน พร้อมจำนวนวัน วันครบกำหนด รอบวางบิล และวงเงินเครดิต (ถ้ามี)
- หลังจบรอบส่งทั้งหมดของวัน สร้างสรุปยอดรายร้านจากรายการส่งทุกครั้งของวันแบบ idempotent
- หน้ารายการเก็บเงินรอบสุดท้าย แสดงเฉพาะร้านเก็บท้ายวัน ยอดที่ต้องรับ และสถานะรับเงิน
- ร้านเก็บท้ายวันที่ปิดก่อนเก็บเงินต้องเปลี่ยนยอดเป็นค้างชำระไปวันถัดไป โดยไม่เปลี่ยนยอดขายเดิม
- บันทึกรับเงินเต็มจำนวน/บางส่วน วิธีชำระ ผู้รับเงิน และคงยอดส่วนต่างเป็นยอดค้าง
- บันทึกเงินจากผู้มารับน้ำแข็งหน้ารถแยกจากยอดร้าน โดยรองรับสถานะจ่าย/ไม่จ่าย และจำนวนเงินจริงเมื่อจำเป็นต้องตรวจเงิน
- พิมพ์ใบสรุปยอด/ใบแจ้งยอดก่อนรับเงิน และออกใบเสร็จรับเงินหลังบันทึกรับเงินจริง โดยพิมพ์ซ้ำแล้วไม่สร้างเลขที่ใหม่
- หน้าลูกหนี้ร้านเครดิต แสดงยอดค้าง รอบวางบิล วันครบกำหนด และประวัติรับชำระ
- รองรับเครื่องพิมพ์ที่เลือกใช้จริง โดยเริ่มจากเอกสาร HTML/PDF ขนาดกระดาษที่ยืนยันแล้ว

เกณฑ์รับงาน: ยอดเอกสารตรงกับรายการส่งและราคา snapshot, ร้านจ่ายทันทีและร้านเก็บท้ายวันอยู่ในคิวที่ถูกต้อง, ร้านที่ไม่ได้รับสิทธิ์ไม่สามารถใช้เครดิต, ยอดค้างข้ามวันไม่สูญหาย และการรับเงิน/พิมพ์ซ้ำไม่ทำให้ยอดหรือเลขที่เอกสารซ้ำ

## Phase 4 — PWA และออฟไลน์

**เป้าหมาย:** ให้ใช้งานได้ต่อเนื่องในจุดที่สัญญาณไม่เสถียร

ขอบเขต:

- ติดตั้งแอปจากเบราว์เซอร์ได้และมีหน้าจอเริ่มต้นแบบ PWA
- เก็บข้อมูลที่จำเป็นของรอบและร้านไว้ในเครื่อง
- คิวรายการบันทึกระหว่างออฟไลน์และซิงก์อัตโนมัติเมื่อออนไลน์
- ใช้ idempotency key และกลไก retry เพื่อกันข้อมูลซ้ำ
- แสดงสถานะออนไลน์/ออฟไลน์ สถานะคิว และข้อผิดพลาดที่ต้องดำเนินการ
- ป้องกันการกดยืนยันซ้ำ พร้อมเสียงหรือสั่นเมื่อบันทึกสำเร็จ (หากอุปกรณ์รองรับ)

เกณฑ์รับงาน: บันทึกรายการขณะออฟไลน์ได้ และเมื่อออนไลน์ข้อมูลถูกซิงก์ครบโดยไม่ซ้ำ

## Phase 5 — รายงาน การตรวจสอบ และการปรับใช้จริง

**เป้าหมาย:** ให้เจ้าของใช้ข้อมูลตัดสินใจ และระบบพร้อมใช้งานจริงระยะยาว

ขอบเขต:

- รายงานยอดส่งรายวัน รายร้าน รายอาคาร รายรอบ และรายประเภทน้ำแข็ง
- รายงานสต๊อกเคลื่อนไหวแยกรถ ทีม รถเล็ก จุดปฏิบัติงาน และถังสำรอง
- รายงานยอดสั่งเริ่มต้น ยอดสั่งเพิ่มระหว่างวัน การใช้ตามยอดขาย ของคืน ของรับเพิ่ม ของเสีย/ละลาย ยอดคงเหลือตามระบบ ยอดนับจริงสิ้นวัน ยอดส่งกลับโรงงาน และส่วนต่างตามจุดที่ตรวจพบ พร้อมระบุว่าไม่มียอดตรวจนับรับจริงจากโรงงาน
- รายงานยอดขาย ยอดรับเงินจริง ยอดค้าง และลูกหนี้เครดิต แยกตามร้านและวันครบกำหนด
- รายงานส่วนต่างของรอบที่ปิดแล้ว
- หน้าประวัติการแก้ไขและการยกเลิก พร้อมตัวกรองวันที่ ผู้ใช้ ร้าน และรอบ
- ส่งออกข้อมูลสำหรับบัญชีหรือการตรวจสอบ
- ทดสอบสิทธิ์ การกันข้อมูลข้ามบทบาท และกรณีซิงก์ผิดพลาด
- ทดลองใช้กับการส่งจริง ปรับขนาดปุ่ม ลำดับข้อมูล และข้อความแจ้งตาม feedback

เกณฑ์รับงาน: แอดมินตรวจสอบย้อนหลังได้ครบ รายงานตรงกับข้อมูลรายการส่ง และผู้ใช้หน้างานใช้งานได้โดยไม่ต้องพึ่งบัตรกระดาษ

## ลำดับการเปิดใช้งาน

เริ่มทดลองกับอาคารเดียวหลังจบ Phase 2 เพื่อทดสอบความเร็วในการบันทึกจริง จากนั้นเพิ่มการตรวจยอดของหัวหน้าใน Phase 3 และทดลอง flow เก็บเงิน/พิมพ์เอกสารใน Phase 3.5 ก่อนเปิดใช้แบบเต็มรูปแบบหลังผ่านการทดสอบออฟไลน์ใน Phase 4

Phase 5 เป็นช่วงยกระดับข้อมูลและความมั่นคงของระบบ โดยสามารถทยอยทำรายงานที่จำเป็นก่อน แล้วค่อยขยายรายงานเชิงวิเคราะห์ตามการใช้งานจริง
