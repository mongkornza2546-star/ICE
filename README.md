# ระบบจัดส่งน้ำแข็งศูนย์ราชการ

ระบบนี้ใช้ควบคุมการสั่งน้ำแข็งจากโรงงาน การเคลื่อนย้ายสต๊อกจากรถบรรทุกไปยังทีมส่ง อาคาร รถเล็ก และถังสำรอง รวมถึงการส่งร้าน การตั้งราคารายร้าน และการติดตามรับชำระ

## หลักการของระบบ

- รถบรรทุกเป็นคลังสต๊อกกลางเคลื่อนที่ของศูนย์ราชการ
- อาคาร จุดปฏิบัติงาน รถเล็ก ทีมส่ง และถังสำรองเป็นจุดถือครองสต๊อกย่อย
- หัวหน้าบันทึกเฉพาะการย้ายสต๊อกที่ตรวจนับได้ เช่น รถไปจุดสต๊อกย่อย การจ่ายเพิ่ม และการคืนรถ โดยระบุต้นทาง ปลายทาง ชนิด จำนวน ผู้บันทึก และเวลา
- การหยิบน้ำแข็งจากจุดสต๊อกย่อยไปส่งแต่ละร้านไม่ต้องให้หัวหน้าบันทึกซ้ำ ระบบหักจากรายการขายของร้านที่ผูกกับจุดนั้นโดยอัตโนมัติ
- การจ่ายน้ำแข็งออกจากรถไม่ถือว่าเป็นยอดขายทันที เพราะน้ำแข็งอาจถูกส่งร้าน คืนรถ เก็บในถังสำรอง หรือเสียหายภายหลัง
- ยอดตัดสต๊อกจริงและยอดที่ใช้คิดเงินร้านต้องแยกจากกัน เพราะการขายอาจเป็นเต็มถุงหรือครึ่งถุง และอาจใช้น้ำแข็งเพิ่มจากการละลาย
- รายการที่บันทึกแล้วใช้การแก้ไขหรือยกเลิกพร้อม audit log ห้ามลบประวัติจริง

## ลำดับงานของหัวหน้ารอบ

### การส่งเป็นรอบ

การส่งน้ำแข็งหนึ่งวันมีรอบมาตรฐาน 5 รอบ และสามารถเพิ่มรอบใหม่ภายหลังได้เมื่อจำเป็น หัวหน้ามีหน้าที่ตรวจนับและบันทึกน้ำแข็งที่จ่ายออกในแต่ละรอบ แต่ไม่ต้องกำหนดว่าพนักงานคนใดต้องเดินอาคาร โซน หรือร้านใด

- พนักงานจัดลำดับการเดิน เลือกอาคาร โซน และร้านที่จะไปในแต่ละรอบเอง
- ร้านเดียวสามารถได้รับน้ำแข็งหลายรอบในวันเดียวกัน หรือไม่รับน้ำแข็งเลยในบางรอบก็ได้
- การไม่รับน้ำแข็งในรอบหนึ่งเป็นเหตุการณ์ปกติ เช่น ร้านขายไม่ดีหรือน้ำแข็งจากรอบก่อนยังไม่หมด ไม่ถือเป็นปัญหาและไม่สร้างยอดขาย
- หากร้านไม่รับน้ำแข็งในรอบนั้น พนักงานไม่ต้องกดสถานะหรือบันทึกอะไร ระบบทราบเพียงว่ารอบนั้นไม่มีรายการขายของร้าน
- รายการขายทุกครั้งต้องผูกกับรอบส่ง ร้าน และจุดสต๊อกต้นทาง
- ยอดขายหน้าบัตรร้านแสดงทั้งประวัติแยกรอบและยอดรวมทั้งวัน
- การจ่ายน้ำแข็งจากรถให้ทีมและการรับเพิ่มระหว่างทางต้องระบุรอบที่เกิดรายการ
- เมื่อจบรอบ หัวหน้าตรวจความคืบหน้าและยอดคงเหลือของรอบได้ แต่ยังไม่ปิดสต๊อกประจำวันหากมีรอบถัดไป
- การนับจริงและส่งน้ำแข็งที่เหลือกลับโรงงานทำหลังจบทุกรอบของวัน

### 1. สั่งน้ำแข็งจากโรงงาน

หัวหน้าบันทึกจำนวนที่สั่งจากโรงงานแยกตามชนิด แต่ไม่ได้อยู่ตรวจนับขณะโรงงานนำน้ำแข็งขึ้นรถ ระบบจึงใช้จำนวนที่สั่งเป็นยอดตั้งต้นโดยสมมุติว่าโรงงานส่งมาครบ

ระบบไม่สามารถสรุปได้ว่าน้ำแข็งขาดหรือเกินจากโรงงาน หากพบส่วนต่างตอนจบรอบต้องแสดงเป็นส่วนต่างที่ยังไม่ทราบสาเหตุ ห้ามระบุว่าเกิดจากโรงงานโดยไม่มีการนับรับจริง

เมื่อรับของแล้ว:

`สต๊อกรถเริ่มต้นตามระบบ = จำนวนที่สั่งจากโรงงาน`

หากน้ำแข็งไม่พอระหว่างวัน หัวหน้าสามารถสั่งเพิ่มจากโรงงานได้หลายครั้ง โดยบันทึกเวลา ชนิด และจำนวนที่สั่งเพิ่ม เมื่อของมาถึง ระบบเพิ่มจำนวนที่สั่งครั้งนั้นเข้าสต๊อกรถตามสมมุติฐานเดิมว่าโรงงานส่งครบ เนื่องจากไม่มีผู้ตรวจนับตอนขึ้นรถ

### 2. จุด Skywalk

เมื่อรถถึง Skywalk หัวหน้าบันทึกการโอนน้ำแข็งจากรถไปยังทีม Skywalk แยกตามชนิดและจำนวน

เมื่อทีมกลับมาที่รถ อาจเกิดรายการต่อไปนี้ได้มากกว่าหนึ่งรายการ:

- คืนน้ำแข็งเหลือจากทีมกลับเข้ารถ
- รับน้ำแข็งเพิ่มจากรถ
- เก็บน้ำแข็งไว้ในถังสำรอง Skywalk
- บันทึกน้ำแข็งเสียหายหรือละลาย
- บันทึกน้ำแข็งที่ส่งให้ร้านตามรายการส่งจริง

### 3. จุดตึก A

ใช้หลักเดียวกับ Skywalk: รถจ่ายของให้ทีมตึก A ทีมเดินส่งตามร้านและโซนย่อย จากนั้นคืนของ รับเพิ่ม เก็บในถังสำรอง หรือบันทึกความเสียหายตามเหตุการณ์จริง

หัวหน้าไม่จำเป็นต้องทราบหรือบันทึกทุกครั้งว่าพนักงานหยิบน้ำแข็งชนิดใดออกจากจุดสต๊อกย่อยตึก A ไปเท่าไร ระบบคำนวณการใช้จากรายการขายของร้านในตึก A โดยร้านและรายการขายต้องผูกกับจุดสต๊อกต้นทางที่ถูกต้อง

ตัวอย่างยอดคงเหลือตามระบบของจุดตึก A:

`ของที่รับจากรถ + ของที่รับเพิ่ม − จำนวนตัดสต๊อกจากรายการขายร้านตึก A − ของคืนรถ − ของเสียหาย = ยอดคงเหลือตามระบบ`

### 4. จุดตึก B และการแยกทีมไปตึก C

เมื่อรถถึงตึก B หัวหน้าแยกการจ่ายสต๊อกอย่างน้อยสองทาง:

- รถบรรทุกไปยังทีมที่ส่งร้านในตึก B
- รถบรรทุกไปยังรถเล็กและพนักงานที่ไปตึก C

หัวหน้าบันทึกผู้รับ จุดหมาย ชนิด และจำนวนน้ำแข็งของแต่ละทีมแยกกัน พนักงานรถเล็กสามารถกลับมารับของเพิ่มเพื่อนำไปเติมที่ตึก C, Skywalk หรือตึก A ได้ โดยทุกครั้งต้องเป็นรายการโอนใหม่ ห้ามเขียนทับยอดเดิม

## อาคารและโซนย่อย

อาคารหนึ่งแห่งมีหลายโซนย่อยได้ เช่น ตึก B มีโดม 1–4, ศูนย์อาหารฝั่ง Tops และศูนย์อาหารฝั่งธนาคาร ร้านต้องผูกกับอาคารและโซนย่อยเพื่อให้พนักงานค้นหาและเลือกเดินได้ แต่ระบบไม่บังคับลำดับเดินตายตัว

ถังสำรองต้องระบุจุดตั้งและมียอดคงเหลือของตนเอง เพื่อให้ทราบว่าน้ำแข็งอยู่บนรถ อยู่กับทีม หรือถูกเก็บไว้ที่จุดใด

## ร้านค้า ราคา และถังน้ำแข็ง

- ร้านในระบบหมายถึงร้านค้าจริงที่อยู่ตามอาคารและโซน ไม่รวม รปภ. หรือบุคคลที่เดินมารับน้ำแข็งจากรถ
- ร้านแต่ละร้านใช้ UUID เป็นตัวตนหลักภายในฐานข้อมูล UUID ต้องไม่เปลี่ยนตลอดอายุของร้านและไม่จำเป็นต้องแสดงให้ผู้ใช้เห็น
- รหัสร้านภายในที่พนักงานใช้ค้นหาและเรียกร้านให้รันลำดับใหม่แยกตามตึก โดยใช้คำนำหน้าตึกที่ไม่ซ้ำกัน เช่น ร้านแรกของตึก A ใช้ `AA01` และร้านแรกของตึก B ใช้ `BB01` รหัสร้านภายในต้องไม่ซ้ำกันทั้งระบบ
- รหัสร้านจากศูนย์ราชการต้องเก็บแยกจาก UUID และรหัสร้านภายใน เพราะบางร้านอาจไม่มีรหัสจากศูนย์ราชการ
- หากร้านย้ายตึก UUID ของร้านต้องคงเดิม ส่วนรหัสร้านภายในสามารถเปลี่ยนตามตึกใหม่ได้ โดยเอกสาร รอบส่ง และรายการย้อนหลังต้องเก็บ snapshot รหัสร้านที่ใช้จริงในขณะนั้น
- หัวหน้าหรือแอดมินเป็นผู้ตั้งราคาของแต่ละร้าน แยกตามชนิดน้ำแข็งและรูปแบบการขาย
- ร้านในโซนเดียวกันสามารถมีราคาไม่เท่ากันได้
- ราคาที่ใช้จริงต้องถูก snapshot ไว้กับรายการขาย เพื่อให้การเปลี่ยนราคาในอนาคตไม่กระทบยอดย้อนหลัง
- ร้านอาจมีถังของตัวเองหรือเช่าถังจากผู้ขาย ระบบต้องเก็บเจ้าของถังและสถานะการเช่าแยกรายร้าน

รูปแบบการขายที่ต้องรองรับอย่างน้อย ได้แก่ เต็มถุงและครึ่งถุง การบันทึกหนึ่งครั้งต้องแยก:

- จำนวนที่ใช้คิดเงิน
- จำนวนที่ตัดสต๊อกจริง
- จำนวนเสียหายหรือละลายและเหตุผล (ถ้ามี)

ตัวอย่างเช่น ร้านถูกคิดเงินเป็นน้ำแข็งเต็มถุง แต่ระหว่างใช้งานอาจต้องเทน้ำแข็งเพิ่มจากถุงอื่นเพราะน้ำแข็งละลาย ระบบต้องแยกจำนวนที่ขายออกจากจำนวนที่นำออกจากสต๊อกจริง โดยไม่เปลี่ยนหน่วยขายของร้าน

## น้ำแข็งสำหรับ รปภ. หรือบุคคลที่มารับจากรถ

น้ำแข็งส่วนนี้ต้องถูกแยกออกจากสต๊อกรถเป็นกองหรือสต๊อกย่อยก่อนเริ่มแจก/ขาย จึงไม่ตัดสต๊อกรถซ้ำทุกครั้งที่มีคนมารับ

รายการรายคนเก็บเพียง:

- มีผู้มารับ
- ชนิดน้ำแข็ง
- จ่ายหรือไม่จ่าย
- จำนวนเงินและวิธีชำระ (เมื่อจำเป็นต้องตรวจยอดเงิน)

ไม่บังคับบันทึกปริมาณรายคน เพราะแต่ละคนได้รับไม่เท่ากัน การควบคุมปริมาณใช้ยอดที่แยกไว้ ยอดเหลือจริง และยอดเสียหายตอนจบ:

`น้ำแข็งที่แยกไว้ − น้ำแข็งที่เหลือ − น้ำแข็งเสีย/ละลาย = น้ำแข็งที่ถูกนำไปทั้งหมด`

รายการนี้ไม่ผูกกับร้านและไม่เข้าเส้นทางส่งร้านค้า

## เงื่อนไขการชำระเงินของร้าน

หัวหน้าหรือแอดมินตั้งเงื่อนไขเฉพาะร้านได้ดังนี้:

1. จ่ายทันที
2. เก็บท้ายวัน
3. เครดิตตามจำนวนวันที่อนุมัติ เช่น 5 วัน
4. วางบิลหรือชำระสิ้นเดือน

เครดิตไม่ใช่ค่าเริ่มต้นและไม่ใช่สิทธิ์ของทุกร้าน ต้องเปิดให้เฉพาะร้านพร้อมกำหนดจำนวนวัน วันครบกำหนด รอบวางบิล และวงเงินเครดิต (ถ้ามี)

ร้านที่ปกติเก็บท้ายวันแต่ปิดก่อนรับเงินต้องคงยอดเดิมเป็นยอดค้างไปวันถัดไป การรับเงินรองรับเงินสด โอน เต็มจำนวน และบางส่วน โดยไม่เขียนทับยอดขายเดิม

## สูตรตรวจสต๊อก

สต๊อกรถ:

`ยอดสั่งเริ่มต้น + ยอดสั่งเพิ่มระหว่างวัน + ของคืนเข้ารถ − ของโอนออกจากรถ − ของเสียบนรถ = สต๊อกรถคงเหลือตามระบบ`

สต๊อกทีมส่งหรือรถเล็ก:

`รับจากรถ + รับเพิ่ม − ใช้ส่งร้าน − คืนรถ − เก็บถังสำรอง − เสียหาย = ของที่ควรเหลือ`

สต๊อกถังสำรอง:

`ของโอนเข้าถัง − ของนำออกไปใช้ − ของเสียหาย = ยอดคงเหลือในถัง`

## การปิดสต๊อกสิ้นวัน

ระบบต้องแสดงน้ำแข็งคงเหลือตามระบบแยกตามชนิด ทั้งยอดรวมของวันและยอดแยกตามจุดที่ยังถือครอง เช่น รถบรรทุก รถเล็ก และถังสำรอง

เมื่อจบวัน หัวหน้าต้องมีช่องกรอก “ยอดนับจริงสิ้นวัน” แยกตามชนิดและจุดที่นับ ระบบเก็บทั้งสองค่าโดยไม่เขียนทับกัน:

- ยอดคงเหลือตามระบบ
- ยอดที่หัวหน้านับได้จริง
- ส่วนต่าง = ยอดนับจริง − ยอดตามระบบ
- หมายเหตุประกอบส่วนต่าง (ถ้ามี)

ยอดรวมตามระบบของน้ำแข็งแต่ละชนิดคำนวณจาก:

`ยอดสั่งเริ่มต้น + ยอดสั่งเพิ่ม − น้ำแข็งที่ออกจากระบบ − น้ำแข็งเสียหาย = ยอดคงเหลือตามระบบ`

การโอนระหว่างรถ ทีม รถเล็ก และถังสำรองไม่ทำให้ยอดรวมทั้งระบบลดลง แต่เปลี่ยนเฉพาะจุดที่ถือครอง ยอดจะลดเมื่อส่งออกให้ร้าน แจก/ขายหน้ารถ หรือบันทึกเป็นของเสียเท่านั้น

หากยอดนับจริงไม่ตรง ระบบต้องแสดงส่วนต่างโดยไม่เดาสาเหตุ และยังปิดวันได้พร้อมเก็บ snapshot ยอดตามระบบ ยอดนับจริง ผู้ตรวจนับ และเวลาปิดวันไว้ตรวจย้อนหลัง

หลังนับเสร็จ น้ำแข็งที่เหลือทั้งหมดต้องถูกบันทึกว่าส่งกลับโรงงาน ระบบเก็บจำนวนที่ส่งกลับไว้ในประวัติ แล้วปิดสต๊อกประจำวันของรถและทุกจุดเป็นศูนย์

ไม่มีการยกยอดน้ำแข็งคงเหลือไปวันถัดไป วันใหม่เริ่มสต๊อกใหม่จากจำนวนที่สั่งจากโรงงานของวันนั้น

## ขอบเขตระบบปัจจุบัน

โค้ดปัจจุบันรองรับการเข้าสู่ระบบ เปิดรอบด้วยร้านที่เปิดใช้งานทั้งหมด ให้พนักงานเลือกบัตรร้านที่จะไปส่งเอง บันทึกการส่งหรือเหตุส่งไม่ได้ ดูยอดรวมรายร้านของวัน และให้หัวหน้าบันทึกยอดสั่งจากโรงงาน โอนสต๊อกระหว่างจุด บันทึกของเสีย และส่งคืนโรงงานผ่าน ledger ต่อเนื่องทั้งวันแล้ว รายการขายหักจากจุดต้นทางของร้านอัตโนมัติ ส่วนการตรวจนับจริงและปิดสต๊อกสิ้นวัน การตั้งราคาและเครดิตรายร้าน รวมถึงรับชำระ ยังเป็นงานที่ต้องพัฒนาตาม [แผนพัฒนา](docs/development-phases.md)
